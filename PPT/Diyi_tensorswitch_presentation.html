The Name: TensorSwitch

The name TensorSwitch combines two concepts - Tensor from TensorStore, and Switch for conversion.

TensorStore is Google's library that we use as the foundation. It's very powerful for handling multidimensional array data, which is exactly what scientific imaging data is. Such as:

  • Efficient reading and writing of Zarr formats
  • Smart chunk-based operations that don't load entire datasets into memory
  • Solid performance for large-scale data

The Switch part is what we need such as:

  • Switch between formats like N5 and Zarr
  • Switch resolutions by creating multiscale pyramids
  • Switch data structures through rechunking

Think of TensorStore as a powerful engine, and TensorSwitch as the vehicle we built around that engine to actually take us where we need to go with the data.

Understanding Data Formats (2-3 minutes)

Why so many formats?

  • Each format emerged from different communities to solve specific problems
  • Can't just standardize on one - each has unique strengths

N5 - Not HDF5, came from Saalfeld Lab at Janelia:

  • Built for BigDataViewer, needed efficient parallel read/write
  • HDF5 had locking issues that made parallel writing slow
  • So N5 solved this by storing one file per chunk
  • Great for Fiji and BigDataViewer, which is Java-based

Zarr2 - Python-friendly alternative:

  • Since python scientific community needed seamless NumPy, dask, xarray, napari integration
  • So it became the community standard
  • Wide codec support and excellent compression options

Zarr3 - developed from zarr2:

  • Big innovation is sharding - grouping multiple chunks into single files
  • Solves the millions-of-tiny-files problem that kills filesystem performance
  • Expensive on cloud storage without sharding
  • Gives more flexible codec pipeline for advanced compression

IMS - Imaris format:

  • Used for visualization of complex 3D, 4D, even 5D datasets
  • HDF5-based internally with built-in pyramids
  • Challenge is - it needs Imaris or compatible tools
  • Converting to Zarr means any tool can read your data, not just Imaris

ND2 - Nikon microscopy format:

  • Acquisition from Nikon microscopes produces ND2 files
  • Rich metadata: acquisition parameters, channels, Z-stacks, time series, calibration
  • Embedded in proprietary binary format
  • Need to extract and translate to work with open-source tools

TIFF - Universal interchange format:

  • Everyone can read TIFF - widely supported
  • But limited: no native chunking, no pyramids, limited metadata
  • Fine for small images, problematic for 100GB (gigabytes) datasets

Why Chunks and Shards Matter

Chunks: Small pieces of array data (e.g., 128×128×64 voxels) that can be read/written independently without loading entire dataset

Shards (Zarr3): Group multiple chunks into single files, reducing filesystem overhead for millions of chunks

Multiscale Pyramids: Store same data at different resolutions (s0=full, s1=half, s2=quarter) for efficient visualization at any zoom level

Format Metadata Structures (2-3 minutes)

  • Now let's look at how these formats actually store their metadata
  • This is the technical heart of what makes conversion challenging

N5 metadata structure:

  • Uses attributes.json with simple fields: dimensions, blockSize, dataType
  • Each chunk is a separate file - makes parallel read/write really efficient
  • Dozens of processes can write different chunks simultaneously without conflicts

Zarr2 metadata structure:

  • Uses .zarray with zarr_format identifier set to 2
  • shape instead of dimensions, chunks instead of blockSize
  • dtype in NumPy notation like less-than-u2 for little-endian unsigned 16-bit
  • This NumPy integration is why Zarr2 became so popular in Python data science

Zarr3 metadata structure:

  • Modernized everything with zarr.json
  • Has chunk_grid structure and codecs array
  • Sharding codec is the key innovation - such as group 8×8×8 chunks into one shard file
  • Reduces millions of files to thousands - huge for cloud storage and filesystem performance

ND2 metadata structure:

  • Embeds OME-XML metadata inside binary file
  • Includes channel information, Z-stack details, time series, pixel calibration
  • All rich acquisition data from microscope in Nikon's proprietary format
  • TensorSwitch will extract and translate this to OME-Zarr structure, which we will discuss later

IMS metadata structure:

  • Uses HDF5 internally with specific hierarchy: DataSet, ResolutionLevel, TimePoint, Channel, Data
  • Already has pyramids built in
  • Need Imaris or compatible tools to read

TIFF metadata structure:

  • Just has IFD tags in header: Width, Height, BitsPerSample, Compression
  • Very limited compared to modern formats
  • No native support for chunking or multi-resolution pyramids

Chunking is:

  • The fundamental concept across all formats
  • Such as instead of loading 100GB dataset into memory, break into small pieces like 128×128×64 voxels
  • Each chunk might be 10MB(megabytes) - load one, process it, write it out, move to next
  • Makes huge datasets processable on normal computers
  • Enables parallel processing on clusters

Sharding on the other hand:

  • Solves the too-many-files problem
  • 1TB(terabyte) dataset with 1MB(megabytes) chunks = million files - and filesystems hate this
  • Cloud storage charges per request - which is expensive
  • Zarr3 sharding groups chunks such as: 512 chunks per shard file
  • 2000 files instead of million, but can still access individual chunks efficiently via internal index

Multiscale pyramids is:

  • For visualization efficiency
  • Store same data at multiple resolutions: s0 full, s1 half, s2 quarter, etc
  • When zooming out in Neuroglancer, loads s2 instead of downsampling s0 on the fly
  • Much faster viewing experience
  • TensorSwitch creates these pyramids automatically

This structure shows how things organized.

Here's a real example from the LICONN lab.

  • The request was - need microscopy data prepared for WebKnossos visualization

The decision steps I worked through:

  • Step 1: Find the files and check what formats we have
  • Step 2: In this case confirmed was mix of ND2 and IMS files from different microscopes
  • Step 3: Choose Zarr3 for WebKnossos, enable sharding to save storage
  • Step 4: Pick the right tasks - nd2_to_zarr3_s0, ims_to_zarr3_s0, then downsample_shard_zarr3

  • This keeps downsampling reusable for any Zarr3 file
