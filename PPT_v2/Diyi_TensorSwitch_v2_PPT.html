Architecture Overview

Here is the full architecture. The green text marks the structural layers: CLI, Readers / Core / Writers, and TensorStore at the foundation.

  • CLI — when a command comes in, it parses the arguments and routes to the right workflow — single file, batch, pyramid, status check.
  • API layer wraps three classes: TensorSwitchDataset for programmatic use, Readers Factory auto-detects the input format and picks the right reader, and Writers Factory handles the output format.
  • Middle layer — Readers reads in data. Writers output Zarr3, Zarr2, or N5. Core includes converter (moves data chunk by chunk), pyramid (plans and submits the downsampling chain), downsampler (does the actual math), and batch (handles multi-file workflows).
  • Utility layer is shared infrastructure used by every layer above — TensorStore spec builders, OME-NGFF metadata writers, anisotropic pyramid planners, LSF resource calculators.
  • TensorStore at the foundation — every read and every write goes through it. Whether it's a local file, HTTP, S3, or GCS — the code above doesn't need to know. TensorStore abstracts it all away. That's what makes the architecture clean.
  • Data flow on the right: an input file hits the Reader, which calls get_tensorstore_spec() to describe it as a TensorStore array. DistributedConverter iterates over chunks and calls the Writer's create_output_spec() to write them out. The output lands as .zarr2, .zarr3, or .n5. That's the entire pipeline — Reader in, Writer out, TensorStore in the middle holding it all together.
  • Q: Why TensorStore as the foundation instead of zarr-python, xarray, or Dask end-to-end? TensorStore gives a single API for reading AND writing across all formats (Zarr2, Zarr3, N5, Precomputed), with built-in sharding support, async I/O, and transparent local/remote access via kvstore. No other single library does all of that.
  • Q: What happens when input and output chunk sizes don't match? Good question — I'll cover that in the Readers section (readers-deep6, Lazy Loading Pipeline). Short answer: Dask acts as the transparent adapter. The converter iterates over the output chunk grid, and Dask automatically stitches together whatever input chunks overlap each request.
  • Q: Is the API layer actually used programmatically, or is it CLI-only? TensorSwitchDataset is there for programmatic use — you can import it and call it from Python directly without the CLI. But right now the main usage is through the CLI.

Reader Tiers Overview

The idea is simple: always use the fastest reader that can handle the format. If TensorStore knows the format natively, use Tier 1 — zero overhead, pure spec dict. If not, fall back down the tiers until something works.

  • Tier 1 is N5, Zarr2, Zarr3, and Neuroglancer Precomputed. These are TensorStore-native — the reader just returns a spec dict and TensorStore opens the file directly. No data conversion, no Dask, just a dict.
  • Tier 2 is the custom readers — TIFF, ND2, IMS, CZI, HDF5. These wrap data as Dask arrays for lazy loading. No data is read until a chunk is actually needed by the converter.
  • Tier 3 is the BIOIO adaptor. Rather than writing a custom reader for every obscure format, TensorSwitch hands off to the bioio ecosystem — install the right bioio plugin and the format just works. LIF and 20+ more formats covered this way.
  • Tier 4 is the Java fallback via Bio-Formats. 150+ formats. JVM startup is about 30 seconds — it's the last resort that hopes to work for more formats.
  • All readers share the same BaseReader contract. There are 3 required abstract methods:
    • get_tensorstore_spec() — returns a TensorStore spec dict or Dask array wrapper
    • get_metadata() — raw format metadata (XML, HDF5 attrs, etc.)
    • get_voxel_sizes() — physical voxel size in nanometers (x, y, z)
  • 3 concrete defaults can be overridden — for example, CZIReader overrides get_ome_metadata() to handle the V-to-T axis rename.
  • get_source_info() — returns a unified dict covering shape, dtype, dimension names, chunk shape, compression type, and memory order. This is what lets the converter auto-configure output settings without knowing which reader is being used.
  • supports_remote() is False by default because most Tier 2–4 readers work only with local files. Tier 1 readers override this to True because TensorStore natively handles s3://, gs://, and http:// paths.
  • Q: Why not just use Bio-Formats for everything? 30-second JVM startup, slower I/O, and it silently loses data in some cases — for example, the CZI multi-view case study (readers-deep5) shows Bio-Formats dropping 35 of 36 views without any warning.

Tier 1: Native TensorStore Readers

Tier 1 is the only tier where TensorStore does all the work. The reader just builds a spec dict and hands it over — no Python reads, no numpy, no Dask. TensorStore's C++ engine opens the file and handles all I/O.

  • The driver names in the spec — n5, zarr3, zarr2, neuroglancer_precomputed — are TensorStore's own built-in format backends. We didn't write N5 or Zarr parsers. We just tell TensorStore which driver to use and where the data is, and it handles the rest. This is the whole reason Tier 1 is zero overhead.
  • TensorStore has four core concepts. Driver is the format backend. kvstore is the storage backend — file, GCS, S3, or HTTP. context controls shared resources like thread pools and concurrency limits. Transaction is the atomic write group for concurrent-safe chunk writes.
  • Each Tier 1 reader has its own path-to-kvstore helper. N5 uses _parse_remote_url() defined in n5.py — it checks the path scheme and returns the right kvstore dict: file for local paths, http for HTTP, etc. Zarr and Precomputed use build_kvstore() from base.py, which does the same thing.
  • The kvstore is the only thing that changes between local and remote — the driver stays 'n5', 'zarr3', etc. That's why Tier 1 supports remote data with no extra code: TensorStore handles the actual I/O once it sees the kvstore spec.
  • ZarrReader detects version by checking which metadata file exists — zarr.json means v3, .zarray means v2. That detection decides which spec to return. Zarr3 gets driver: 'zarr3', Zarr2 gets driver: 'zarr'. One character difference in the spec, but a completely different TensorStore backend.
  • The sharding difference is the big one. Zarr3's zarr3 driver reads the sharding_indexed codec straight from zarr.json and handles it transparently — outer chunks contain multiple inner chunks, each zstd-compressed by default. Zarr2's zarr driver has no sharding concept at all. You don't write any sharding logic — TensorStore handles it once it sees the codec in the metadata.
  • Multi-scale is handled differently by each format. Zarr3 and Zarr2 use a dataset_path joined into the kvstore path — so s0 becomes part of the file path. N5 puts the dataset path into spec['path'] separately. Precomputed uses scale_index as a number directly in the spec — no path joining at all.
  • PrecomputedReader is for Neuroglancer's precomputed format, common at Janelia for large segmentation volumes. scale_index=0 is the highest resolution, scale_index=1 is 2x downsampled, and so on. TensorStore picks the right scale from the info file automatically.
  • Q: What does 'open': True do in the spec? It tells TensorStore to open and validate the dataset immediately rather than lazily. This catches missing files or corrupt metadata up front instead of at first read.
  • Q: How does data_copy_concurrency from LSB_DJOB_NUMPROC work? LSB_DJOB_NUMPROC is set by LSF to the number of cores allocated to the job. TensorStore's context uses it to cap its internal thread pool size, preventing the job from exceeding its allocation and getting OOM-killed by the cluster.

Tier 2/3 Lazy Loading Pipeline

Without loading the whole file into memory. Dask is the key adapter that separates input chunk boundaries from output chunk boundaries, so they don't need to match.

  • Start with the pipeline diagram on the top-left. For TIFF: tifffile.imread with aszarr=True creates a virtual Zarr store — it maps TIFF pages to Zarr chunks without reading any pixel data. da.from_zarr wraps that in a Dask task graph holding only metadata — about 1 megabyte (MB) for a 200 gigabyte (GB) file. Nothing has been loaded yet at this point. The reader then wraps the Dask array in a dict with driver='array'. This is NOT a real TensorStore spec — it's just a marker dict so it can go through the same get_tensorstore_spec interface as Tier 1 readers. TensorStore never opens it. The converter detects driver=='array', extracts the Dask array directly, and sets input_store to None. TensorStore is completely bypassed on the read side.
  • Below the diagram — the 3-step flow of how chunk boundary mismatch is handled. The converter iterates over the OUTPUT chunk grid, not the Dask chunk grid. Dask's .compute() acts as the transparent adapter between mismatched input and output boundaries.
  • Step 1 — Output chunks are the master. get_chunk_domains() in chunk_utils.py (lines 67–94) tiles the entire output space into output-sized chunks — for example, 128 cubed. It is completely independent of the input Dask array's chunk boundaries. The output writer decides the iteration order and chunk size. This completely separates input format from output format.
  • Step 2 — Convert each output chunk domain to Python slices. _domain_to_slices() in converter.py (lines 522–537) turns a TensorStore IndexDomain into standard Python slices — like (slice(0,128), slice(0,128), slice(0,128)). These are regular Python slices that work on any array, including Dask arrays.
  • Step 3 — Dask handles the stitching. self._dask_array[slices].compute() at converter.py (lines 343–350) is where the magic happens. Dask internally figures out which of its own chunks overlap the requested slice. It loads only the overlapping portions from disk. It automatically stitches them into a single contiguous numpy array. And it returns it — the converter never knows or cares about Dask's internal chunk boundaries. There is no explicit chunk mapping anywhere in the code.
  • The top-right comparison table shows the key difference between Tier 1 and Tier 2/3. Tier 1 readers return real TensorStore specs — the converter calls ts.open() and uses TensorStore's native IndexDomain. TensorStore manages both input and output chunks. Tier 2/3 readers bypass TensorStore for reading — Dask manages input chunks, TensorStore manages output chunks, and .compute() is the bridge between them.
  • The middle table shows the lazy mechanism per format. Each format produces a Dask task graph, not actual data. CZI uses dask.delayed per plane. ND2 uses nd2.to_dask(). IMS and HDF5 wrap h5py datasets with da.from_array(). The converter treats them all identically — iterate output chunks, slice, .compute(), write.
  • Here is an example — Say input is a TIFF with 512×512 page-based Dask chunks, and output is Zarr3 with 128 cubed chunks. Request [0:128, 0:128, 0:128] — Dask recognizes one internal chunk covers this, loads only that portion of the TIFF page, done. Request [384:512, 384:512, 384:512] — if it crosses a page boundary, Dask loads 2 or more pages, stitches the overlapping region, returns a clean 128 cubed numpy array. The converter code is identical in both cases — just calls .compute() and gets back a numpy array. Output chunks don't need to know about input chunks — Dask is the adapter layer.
  • Memory efficiency. Each .compute() call loads only the requested slice. Peak memory is one output chunk — about 32 MB — not the whole file. This is why TensorSwitch can convert 200 GB TIFFs on cluster nodes with only 15 GB of RAM. Per-chunk TensorStore transactions on the write side ensure memory doesn't build up during writes either.
  • Future consideration: ts.virtual_chunked? We currently don't use TensorStore's virtual_chunked anywhere — Dask handles all the lazy chunk stitching. But virtual_chunked could be a future enhancement. It lets you create a TensorStore array backed by a Python callback function — so instead of Dask managing input chunks, TensorStore itself would call your function to fill each chunk on demand. This could unify the Tier 1 and Tier 2/3 read paths into a single TensorStore-native pipeline. Right now the Dask approach works well and is simpler, but virtual_chunked is worth exploring if we want tighter TensorStore integration on the read side.
  • Q: Why Dask instead of ts.virtual_chunked for Tier 2/3 readers? Every microscopy library (tifffile, nd2, bioio) already produces Dask arrays natively. Using virtual_chunked would require writing custom per-format callbacks that replicate what Dask already does. The tradeoff is TensorStore can't manage input chunks for non-native formats, but Dask's .compute() handles chunk boundary stitching transparently. Migrating to virtual_chunked would unify Tier 1 and Tier 2/3 read paths but adds complexity with no clear performance win today.
  • Q: Peak memory is "one output chunk ~32 MB" — what's the actual process memory? The 32 MB refers to data only. Actual process memory is: tifffile index (~1 MB for a 200 GB file) + one Dask computation result (~32 MB) + one TensorStore write buffer (~32 MB) + Python/TensorStore overhead (~2–4 gigabytes) = roughly 4–5 GB peak. The 14 GB seen in the CZI benchmark is higher because Dask's task graph metadata for 155,000 delayed tasks takes ~2 GB on its own.

Tier 2: Custom Optimized Readers

These five Tier 2 readers cover the main microscopy formats we see often. They all work the same way — each one wraps the data as a Dask lazy array, hands it to TensorStore as driver: 'array', and nothing actually loads until a chunk is requested.

  • TIFF: tifffile has a trick where you can open a TIFF as a zarr store with aszarr=True. That gives a fully lazy, zero-copy view. For a folder of TIFFs, dask-image globs them and stacks them into one array.
  • ND2 is the simplest — the nd2 library has a built-in to_dask() call. One line. The file handle stays open in the background because Dask needs it alive when pulling chunks later.
  • IMS is HDF5 under the hood, so we walk the tree ourselves: DataSet → ResolutionLevel 0 → TimePoint → Channel → Data. Each channel becomes its own Dask array, then we stack them into CZYX.
  • CZI is the most complex. We build a V × C × Z grid of per-plane dask.delayed reads — each plane opens and closes the file on its own, so it's safe to run in parallel across LSF cores. The [:, :, 0] at the end is because pylibCZIrw's read() always returns a 3D array (Y, X, C) where the third dimension is a pixel-level color channel — 1 for grayscale (Gray16), 3 for BGR color. This is by design in pylibCZIrw for consistent dimensionality, similar to OpenCV's (H, W, C) convention. Microscopy data is almost always Gray16, so we get (Y, X, 1) and index with 0 to drop the singleton so Dask can stack the planes cleanly. The code guards this: it only squeezes when ndim == 3 and shape[2] == 1, so it would still work correctly if someone passed BGR data.
  • HDF5 is the most generic. We try the user-specified path first. If nothing's given, we look for common dataset names, and if that fails too, we just take the largest dataset in the file.
  • Q: CZI opens and closes the file for every single plane — isn't that expensive? The file handles are not thread-safe, so they can't be shared across Dask workers. Each open/close is sub-millisecond — CZI uses an internal block index so it's instant. The 446 GB CZI conversion still completed in 23.2 hours with 14 GB peak memory. Keeping one handle open would lose all parallelism.
  • Q: Why write custom Tier 2 readers instead of just using bioio (Tier 3) for these formats? Performance and correctness. tifffile's aszarr gives zero-copy lazy loading — bioio would load pages eagerly. bioio-czi crashes on multi-view CZI files (KeyError on missing scene names). Custom readers give direct control over how data becomes Dask arrays.
  • Q: Why pylibCZIrw instead of bioio-czi or aicspylibczi? pylibCZIrw is Zeiss's official C library wrapper. bioio-czi crashes on multi-view files because it expects a Name attribute on every Scene element — our file only has Index. pylibCZIrw handles all CZI variants correctly.
  • Q: What if tifffile can't open a TIFF as a zarr store (aszarr)? It falls through to Tier 3 (bioio) or Tier 4 (Bio-Formats). The tier system is the fallback chain.
  • Q: ND2 file handle stays open — isn't that a resource leak? It's necessary because Dask lazy arrays reference the underlying file. If you close the handle, .compute() fails later. The handle gets cleaned up when the reader object is garbage collected.

Tier 4 is the thinnest layer in the whole codebase. BioFormatsReader inherits everything from BIOIOReader — the loading logic, the Dask wrapping, the TensorStore spec.   • _get_bioformats_reader() imports bioio_bioformats.Reader — that's the BIOIO plugin backed by the Bio-Formats Java library. It calls _ensure_bioformats_loaded() first to make sure the JAR is in the JVM classpath before anything starts.
  • _ensure_bioformats_loaded() calls bioformats_jar.get_loci(). That downloads the Bio-Formats JAR if it's not already there and registers it with the JVM. The JVM itself starts the first time bioio_bioformats is actually used.
  • Once the reader class is returned, it's passed to BIOIOReader.__init__() as reader=. From that point on everything is identical to Tier 3 — BioImage, dask_data, driver: 'array'. Tier 4 literally is Tier 3 with a Java plugin.
  • For formats that have a native Python plugin in Tier 3, there's no reason to use Tier 4.
  • The formats that make Tier 4 worth it are the ones no Python plugin covers — VSI, OIB from Olympus, SCN from Leica, Volocity, and many legacy vendor formats. If Tier 3 can't open it, try Tier 4 with --use_bioformats.
  • Q: Does Bio-Formats load data lazily or into memory? The Python interface is lazy — it goes through BIOIO's dask_data property. But under the hood, Bio-Formats may buffer entire planes in Java memory. The lazy boundary is between Python and Java — Java-side memory usage is harder to control.

CZI Multi-View Case Study

The source file is a 1.1 terabyte (TB) Zeiss lightsheet CZI with 36 views. Think of each view as a separate camera angle of the same sample. The full data is 5D: 36 views × 2 channels × 2153 Z slices × 1920×1920 pixels. After running it through Tier 2, 3, and 4:

  • Tier 2 is the only one that handles it correctly. load_czi_stack() reads all 36 views at once and puts them into a single Dask array. The chunks are (1, 1, 1, 1920, 1920) — that means each chunk is exactly one full 1920×1920 image plane at a specific view, channel, and Z position. Dask only loads one chunk into memory at a time, so the peak memory stays at 14 GB even though the output is 446 GB. The V axis gets renamed to t so viewers like Neuroglancer can show all 36 views as a time slider — you can click through them.
  • Tier 3 crashes before reading a single byte. The bioio-czi plugin reads the CZI XML metadata and looks for a Name attribute on every Scene element. This file's scenes only have an Index — no Name. That's a KeyError in bioio-czi's own code, not in TensorSwitch. It needs to be fixed upstream in the bioio-czi library.
  • Tier 4 completes but silently loses 35 of the 36 views. Here's why: Bio-Formats maps each CZI view to a separate "series" — think of it as Bio-Formats treating each view as its own dataset. Each series contains all the channels and Z slices for that one view. So series 0 = view 0, series 1 = view 1, and so on up to series 35. The reader defaults to reading series 0 only. That gives you the full data for view 0: both channels, all 2153 Z slices — shape (1, 2, 2153, 1920, 1920) where T=1 instead of T=36. The C=2 is preserved because channels live inside a series. But the V=36 collapses to T=1 because only one series was read. To get all 36 views you need to read series 0 through 35 separately and stitch them together.
  • This test is why a custom Tier 2 CZI reader was written. Neither bioio-czi nor Bio-Formats handled multi-view data correctly out of the box.
  • Q: Could you modify Tier 4 to loop through all 36 series and stitch them? Technically yes, but it would mean reading 36 series sequentially through Java and stitching in Python. pylibCZIrw does this natively in one call — simpler and faster.
  • Q: Why rename V to T instead of keeping the view axis? Neuroglancer and other viewers don't have a native "view" axis. They do understand "time" (T). Renaming V→T lets you use the time slider to browse all 36 views. It's a viewer compatibility workaround.
  • Q: 14 GB peak memory for a 446 GB output — where does the 14 GB come from if each chunk is only ~7 MB? Dask's task graph metadata for 155,000 delayed tasks takes ~2 GB by itself. Plus Python/TensorStore overhead (~2–4 GB), plus write buffers. The 14 GB is the process-level peak, not just data.

Writers Overview

Three formats for three downstream ecosystems.

  • Zarr3Writer uses sharding. Instead of one file per chunk, multiple chunks are packed into a single shard. For a 1 TB dataset that means hundreds of files instead of millions — that matters on S3 and GCS where listing huge numbers of objects is slow and costs money. OME-NGFF v0.5 metadata lives in zarr.json.
  • Zarr2Writer is for tools that haven't moved to Zarr v3 yet — BigStitcher, some napari plugins, and other legacy tools. Metadata goes in .zattrs. The _ARRAY_DIMENSIONS key lets xarray and dask use named dimensions out of the box without extra configuration.
  • N5Writer is for the Java ecosystem at Janelia — BigDataViewer, Paintera, and Fiji plugins that predate Zarr. Metadata is in attributes.json following N5 conventions. (I haven't fully tested the N5 writer yet.)
  • All three implement the same BaseWriter abstract class. DistributedConverter calls create_output_spec() to set up the destination, write_chunk() for each chunk during conversion, and write_metadata() at the end to finalize the OME-NGFF structure.
  • Q: Why not just use Zarr3 for everything? Not all tools support Zarr3 yet. BigStitcher needs Zarr2. Java tools like BigDataViewer and Paintera need N5. The three writers exist because the downstream ecosystem is fragmented.
  • Q: What's the actual file count difference with sharding? For a 2304×2304×906 volume with 1024³ shards and 64³ inner chunks: about 9 shard files vs 19,440 individual chunk files without sharding. That's a 2000× reduction in file count.
  • Q: Why BaseWriter instead of each writer being independent? The abstract interface means DistributedConverter works with any format without knowing which writer it's using. Adding a new output format = implement 3 methods.

Zarr3Writer + BaseWriter

Left side shows the TensorStore spec zarr3_store_spec() builds. The key is what's inside metadata.codecs.

  • sharding_indexed is the outer codec. It groups many inner chunks into one shard file on disk. Without it, a 1 TB dataset can generate millions of files. That's the single biggest win for cloud storage on S3 or GCS.
  • transpose goes INSIDE sharding_indexed's codecs list — that placement is critical. Outside causes TensorStore to throw "incompatible chunk size constraints". F-order means Fortran column-major layout, which Neuroglancer expects.
  • bytes converts the array to raw little-endian bytes — little-endian means the least significant byte comes first in memory, which is the standard byte order on x86/ARM CPUs. This codec is required between the array data and any compression codec like zstd.
  • zstd is the final compression step.
  • The OME-NGFF panel shows the nested folder layout. The live example link opens a file in Fileglancer so you can browse the actual raw/ and labels/segmentation/ structure.
  • BaseWriter — the abstract interface all three writers share. Three steps, three methods:
    • create_output_spec() — runs once up front, before any LSF jobs start. Makes sure the output store exists so workers don't race to create it.
    • write_chunk() — runs per chunk during conversion. Each write is wrapped in a TensorStore transaction, so it's atomic and safe to retry if something fails.
    • write_metadata() — runs once at the very end, after all chunks are confirmed written. Finalizes the OME-NGFF metadata.
  • Q: Why does transpose go inside sharding instead of outside? The inner codecs apply to each inner chunk individually. Transpose outside would try to apply to the shard-level shape, which doesn't match the inner chunk shape constraints — TensorStore rejects it. This was a real bug we hit and fixed.
  • Q: Why zstd over blosc or gzip? zstd has faster decompression than gzip with similar compression ratio. blosc is great for in-memory work but zstd is better for on-disk and cloud storage. It's also the recommended default in the Zarr3 spec.
  • Q: Why little-endian specifically? It's the native byte order of x86 and ARM processors — virtually every machine running TensorSwitch. Using native endianness avoids byte-swapping overhead on read. The Zarr3 bytes codec requires you to explicitly declare the endianness so data is portable across architectures.
  • Q: Why does create_output_spec() need to run before LSF jobs? Race condition prevention. Multiple LSF workers starting at the same time would all try to create the zarr.json and directory structure. Pre-creation ensures the store exists before any worker starts writing.

Zarr2Writer and N5Writer

  • Zarr2Writer uses the "zarr" TensorStore driver, not "zarr3". The compression field is called "compressor" — a single codec dict — instead of "codecs", which is the Zarr3 list. That's the storage-level difference between v2 and v3.
  • dimension_separator is explicitly set to "/" (nested directory structure). Zarr2's default is "." which gives flat filenames like "0.1.2". Setting "/" gives nested folders "0/1/2", which is much better for filesystems and object stores with many chunks.
  • fill_value must be 0, not null. Neuroglancer expects an integer — using null causes display artifacts that only appear in Neuroglancer, not in napari or Python, so it was hard to catch.
  • write_metadata() also writes two things to .zattrs: OME-NGFF v0.4 multiscales with axes and coordinateTransformations, and _ARRAY_DIMENSIONS. The _ARRAY_DIMENSIONS key is a Zarr v2 convention — it stores axis names like ["z", "y", "x"] alongside the OME-NGFF metadata. When you open the file with xarray.open_zarr() or dask, axis 0 is already named "z", axis 1 is "y" — no extra configuration needed.
  • N5Writer uses the "n5" TensorStore driver. Field names come from the original N5 Java spec — "dimensions", "blockSize", "dataType" instead of shape, chunks, dtype. Metadata goes in attributes.json, not zarr.json or .zattrs.

Q: Why does fill_value need to be 0 instead of null for Zarr2?
The Zarr v2 spec allows fill_value: null, and most Python tools (zarr-python, napari, xarray) handle it correctly. But Neuroglancer's JavaScript Zarr reader interprets null as "no fill value" and renders missing chunks as garbage data instead of zeros. This only shows up in Neuroglancer, not in Python tools, making it very hard to catch during testing. For uint16 microscopy data, zero is the natural background value, so fill_value: 0 is universally compatible.

Q: Why zstd for Zarr2 instead of blosc?
Blosc is Zarr2's historical default, but zstd gives better compression ratios at similar speed. TensorStore supports zstd natively, and it's the same codec used in the Zarr3 writer, so using zstd everywhere keeps behavior consistent across all three output formats.

Q: Why gzip for N5 instead of zstd?
The N5 ecosystem is Java-based (BigDataViewer, Fiji, MATLAB). Gzip is the one codec every N5 reader supports out of the box. Zstd and blosc require extra Java native libraries that may not be installed, so gzip is the safe default for maximum compatibility.

Q: What is _ARRAY_DIMENSIONS and why write it alongside OME-NGFF?
_ARRAY_DIMENSIONS is an xarray/dask convention that maps array axes to named dimensions. OME-NGFF stores the same info in multiscales.axes, but xarray.open_zarr() looks for _ARRAY_DIMENSIONS first. Writing both means the data works natively in xarray/dask AND in OME-NGFF-aware viewers like napari.

DistributedConverter

  • DistributedConverter is the conversion engine, which only works with TensorStore specs without knowing data format. The reader gives it an input, the writer gives it an output, and it copies chunk by chunk.
  • The code on the left walks through the four steps. First, open the input — Tier 1 readers like N5 and Zarr give back native TensorStore specs. Tier 2 readers like TIFF and CZI give back a Dask array instead. The converter handles both.
  • Second, create the output by calling the writer's create_output_spec(). The writer handles everything format-specific — sharding, codecs, chunk layout.
  • Third and fourth, iterate over chunks and write each one inside a TensorStore transaction. That means if a job crashes halfway, you never get a corrupted partial chunk in the output. Other jobs writing to different chunks keep going independently.
  • There are two modes. Single-job runs locally or on the cluster with --submit — same code, just wrapped in bsub. Multi-job LSF splits the dataset into chunk ranges with --start_idx and --stop_idx, so you can run 10 or 100 jobs in parallel on different parts of the data.
  • Key features. Transaction-per-chunk is what makes multi-job safe: each chunk write either fully completes or rolls back, so 100+ parallel jobs can write to the same Zarr without corruption.
  • By default the converter preserves source dimensions — 3D in, 3D out. No unwanted singleton axes added. If you need 5D TCZYX output for legacy tools, there's an --expand-to-5d flag, but it's off by default per OME-NGFF RFC-3.
  • It also detects and preserves memory order — if the source is F-order (column-major, like Neuroglancer Precomputed), the output stays F-order. You can override with --force_c_order or --force_f_order.
  • Singleton channel squeeze handles a Precomputed quirk: TensorStore's precomputed driver always adds a 'channel' axis even when there's only one channel, turning a 3D volume into [Z, Y, X, Channel=1]. The converter detects that the channel dimension is size 1, squeezes it out, and writes clean 3D output.

Q: What happens when multiple LSF jobs write to the same shard file simultaneously?
TensorStore's sharding_indexed codec handles this safely — each inner chunk has its own byte-range entry in the shard index, and writes are serialized at the shard level by TensorStore's internal locking and transaction mechanism. The per-chunk ts.Transaction ensures each chunk write is atomic. Two jobs with adjacent chunk ranges may target different inner chunks in the same shard, and TensorStore handles this correctly on POSIX filesystems like /nrs. Could be a problem on object stores without strong consistency.

Q: What happens when a chunk write fails mid-conversion?
The current strategy is warn-and-skip: the exception is printed as a warning and the loop continues. The return dict includes chunks_processed which will be less than total_chunks if any failed. Transaction-per-chunk means partial writes are impossible — a chunk is either fully written or not at all. The rationale is one corrupt source region should not abort a terabyte (TB)-scale conversion. There is no post-hoc verification step yet — a --verify mode that reads back chunk metadata would strengthen this.

Q: Why does the converter need to handle both TensorStore specs and Dask arrays?
Tier 1 readers (N5, Zarr, Precomputed) return native TensorStore specs because TensorStore has built-in drivers for those formats. Tier 2 readers (TIFF, CZI, KLB) have no TensorStore driver, so they load data through their own libraries into Dask arrays. The converter checks spec['driver'] == 'array' to decide which path to take — TensorStore direct read vs Dask .compute().

Q: Why F-order preservation instead of always using C-order?
Neuroglancer Precomputed format stores data in F-order (column-major). If you convert Precomputed to Zarr and silently flip to C-order, the data is transposed when viewed. Detecting and preserving the source order avoids this. Most microscopy formats are C-order, so this only kicks in for Precomputed inputs.

Q: Why is --expand-to-5d off by default?
OME-NGFF RFC-3 recommends preserving source dimensionality — a 3D ZYX volume should stay 3D, not get padded to 5D TCZYX with singleton T=1 and C=1 axes. The padding wastes metadata complexity and confuses viewers that interpret T=1 as "one timepoint" rather than "no time dimension." The flag exists for legacy tools that expect exactly 5D.

Downsampler: How Downsampling Actually Works

  • Three steps to show how downsampling works. The key is step 2 — TensorStore's downsample driver. We don't manually read blocks of voxels and average them. Instead we open a virtual TensorStore view that looks like the downsampled data already exists. "Virtual" here means it's TensorStore's own built-in 'downsample' driver — part of TensorStore's C++ core, no extra library needed. When you read a chunk from it, TensorStore fetches the corresponding source region and reduces it on the fly.
  • Step 3 just copies chunks from the virtual view to the output store, using the same transaction-per-chunk pattern as DistributedConverter. That's why multi-job LSF works here too — different jobs copy different chunk ranges.
  • Six methods on the right. Mean averages voxel intensities — better for microscopy images. Mode picks the most common value in each region — better for segmentation labels, because averaging label 1 and label 2 would give 1.5, which isn't a valid label. Stride just picks every Nth voxel — fastest but introduces aliasing.
  • Auto-detect checks the input path for keywords like 'label', 'mask', 'seg'. If segmentation is detected, it uses mode. Otherwise mean. So running --auto_multiscale on .../labels/segmentation/s0 automatically picks mode without you having to specify it.
  • Cumulative factors. In a pyramid, each level has a per-level factor — how much to reduce from the previous level. For example, s0→s1 is [1,2,2] (skip Z, halve Y and X), and s1→s2 is also [1,2,2]. The cumulative factor multiplies these together: for s2 it's [1, 2×2, 2×2] = [1, 4, 4]. This single cumulative factor tells the downsample driver exactly how much to reduce from s0 in one step. The Downsampler passes this directly to the TensorStore downsample spec so it can read from s0 (or the previous level in chained mode) and produce the correct output in a single pass.

Q: Why use TensorStore's downsample driver instead of scipy.ndimage or skimage?
TensorStore's downsample driver operates at the chunk level in C++ — it reads only the source region needed for each output chunk and reduces it without loading the whole array into Python. scipy/skimage would require the entire array (or large slices) in memory as NumPy arrays. For terabyte (TB)-scale data, that's not feasible.

Q: Why does mode work for segmentation labels but mean doesn't?
Label IDs are categorical, not continuous. If a 2×2 region has labels [1, 1, 2, 3], mean gives 1.75 which doesn't correspond to any label. Mode gives 1, which is the most common label in that region — preserving the semantic meaning. This is critical for neuroscience segmentation data where each label ID maps to a specific neuron or structure.

Q: What happens with anisotropic data where Z resolution is much coarser than XY?
The per-level factors handle this — [1, 2, 2] means "don't downsample Z, halve Y and X." This is common in microscopy where Z spacing might be 1 μm but XY is 0.2 μm. PyramidPlanner (next slide) calculates these factors using Yurii Zubov's algorithm based on actual voxel sizes, so Z only gets downsampled once XY resolution catches up.

  • The chain diagram at the top is the key idea — each level reads from the previous level, not from s0. This is the architecture that makes deep pyramids practical on the cluster.
  • The Python code on the left is the core loop inside _generate_chained_coordinator_script(). It iterates over each level in the pyramid plan. The critical line is where source is set to the previous level — level minus one, not s0. Then it auto-calculates memory based on that source shape. Because the source shrinks at every level, later levels need less memory, less wall time, fewer cores.
  • The bash on the right is what that loop generates — a coordinator script. The submit_and_wait function submits a downsample job via bsub, then calls bwait to block until it finishes. Only then does the next level get submitted. Simple sequential script, no complex LSF dependency graphs.
  • Notice the decreasing resources in the bash — 30 gigabyte (GB) for s1, 15 GB for s2, 8 GB for s3. And the wall time drops too — 1:00, 0:30, 0:15. That's the direct benefit of chaining — each level's source is smaller, so it needs less of everything.
  • The Auto-Calculated Resources panel shows the actual formulas, which we will talk about in more detail later. Memory is based on shard size times the downsample factor — that's how much source data you need to read per output shard — plus write buffer and overhead, with a 1.5× safety margin, clamped between 8 and 128 GB. Wall time uses empirical shard-size tiers: small shards under 0.1 GB take about half a minute each, medium shards up to 1 GB take about 2 minutes, and large shards over 1 GB take about 4 minutes. Multiply by number of shards, add read overhead based on source size, then apply 2× safety margin. The range is 15 minutes to 12 hours. Cores are derived from memory — divide by 15 and double, capped at 8. We cap at 8 because this is I/O-bound work — more cores don't help. The Janelia cluster enforces 15 GB per core minimum, so if you request 6 cores you get at least 90 GB allocated. The code pre-calculates this so our request matches what the cluster actually gives us.
  • Why chain? I first attempted to have all levels read directly from s0 — the fan-out pattern. For s4, that meant 2048× read amplification per output voxel. The s4 job ran over 4 hours and never finished — because all parallel pyramid level jobs were trying to read from s0 simultaneously, each one pulling exponentially more data. I switched to chained, where each level reads only the previous level. That gives constant 8× amplification per level. The CZI dataset — 446 GB output, 6 levels — completed in 73 minutes total.
  • After all levels complete, the coordinator updates the root OME-NGFF metadata with all the pyramid level paths. This runs last to avoid race conditions.

Q: Doesn't chaining introduce compounding quality loss from repeated downsampling?
In theory, all-from-s0 is more mathematically correct. In practice, the quality difference is negligible for microscopy data — averaging 8 voxels (2×2×2 from previous level) versus averaging 2048 voxels (from s0) produces differences below the noise floor of the microscope. The performance difference is not negligible — the CZI dataset completed in 73 minutes chained versus never finishing from s0 due to 2048× read amplification at s4.

Q: What happens if the coordinator script is killed between bwait calls?
Any currently-running LSF job continues to completion — it's independent of the coordinator. But subsequent levels are never submitted. The data already written is valid — each level is a complete, self-contained Zarr array. To recover, re-run --auto_multiscale with --skip_existing, which checks for existing zarr.json files and only submits missing levels. The current bwait approach was chosen over LSF job dependencies (bsub -w "done(JOB_ID)") for simplicity and debuggability — you can watch the coordinator's stdout in real time.

Q: Why bwait instead of LSF job dependencies (bsub -w "done(JOB_ID)")?
bwait blocks the coordinator script until the job finishes, so the coordinator runs sequentially and its stdout shows real-time progress. LSF job dependencies (bsub -w) would submit all jobs upfront with dependency chains, but then you can't see which level is running, and if one fails, the rest silently pend forever. bwait is simpler to debug — you just tail the coordinator log.

Q: Why cap cores at 8 if memory could be 128 GB?
Downsampling is I/O-bound — TensorStore reads source chunks and writes output chunks, but the CPU work (averaging voxels) is trivial. Adding more cores doesn't speed up disk reads. The 8-core cap avoids wasting cluster slots. At 128 GB with 8 cores, you get 16 GB/core which satisfies Janelia's 15 GB/core minimum.

Q: How does the pyramid know when to stop adding levels?
PyramidPlanner uses three stopping rules: (1) total array size drops below one chunk's worth of bytes, (2) all spatial dimensions fall below the chunk shape threshold, (3) cumulative magnification drifts more than 4% from a power-of-2 (for WebKnossos compatibility).

Q: What is Yurii Zubov's anisotropic algorithm?
It calculates per-level downsample factors based on actual voxel sizes. If Z spacing is 1 μm and XY is 0.2 μm, it only downsamples XY until the effective resolution in all axes is similar, then starts downsampling Z too. This prevents over-smoothing the coarser axis.

Q: How does the batch converter handle heterogeneous files — mixed formats, mixed resolutions?
Each file gets its own auto-detection pass through the reader factory. The glob pattern filters by extension, but within that, each file is opened independently to determine shape, dtype, voxel sizes, and chunk configuration. If a file can't be opened (corrupt, unsupported), that job index fails and logs the error while all other jobs continue. The --status command shows which files failed. However, all files in a batch share the same --output_format, --compression, and --chunk_shape CLI arguments — there is no per-file override. For directories with genuinely different output needs, run separate batch commands.

Q: Why a TSV index file instead of passing paths directly to each job?
LSF job arrays share one bsub command — you can't pass different arguments to each array element. The TSV index file is the standard workaround: all jobs get the same command but each reads a different row based on LSB_JOBINDEX. It also serves as a persistent record of what was submitted, useful for debugging and for check_status.

Q: Why %100 concurrency limit instead of running all 1890 jobs at once?
Two reasons: (1) storage I/O bandwidth — /nrs can handle maybe 100 simultaneous write streams before throughput drops, and (2) LSF scheduler fairness — submitting 1890 runnable jobs monopolizes the cluster queue. The %100 limit keeps throughput near maximum while leaving cluster capacity for others.

Q: Why not use Dask distributed or Spark instead of LSF job arrays?
LSF job arrays are simpler and more robust for file-level parallelism. Each job is fully independent — no coordination, no shared state, no driver process. If one crashes, others are unaffected. Dask distributed adds a scheduler process that becomes a single point of failure, and Spark's JVM overhead is unnecessary for what's essentially "run the same command on 1890 different files."

tensorstore_utils.py: TensorStore Spec Builder

  • tensorstore_utils builds TensorStore specs. Every writer calls into here — Zarr3Writer, Zarr2Writer, N5Writer.
  • Something worth noting. When we use F-order (Fortran/column-major) with sharding, the transpose codec has to go INSIDE the sharding_indexed codec's inner_codecs list — before bytes and compression. If we put it at the array level instead, TensorStore throws an "incompatible chunk size constraints" error. This was a real debugging headache for me before because the error message doesn't tell you what's wrong.
  • For NON_SPATIAL_AXES and build_default_shape(). By default, channels, time, and view axes get chunk size 1. The reason is most microscopy workflows view one channel or one time point at a time — if chunk size is 1, loading a single time point reads exactly one chunk. If it were 5, you'd pull in 5 time points just to see one. Spatial axes (z, y, x) get a larger size, typically 64 for inner chunks and 1024 for shards. This means the same spec-building code works whether you're writing to /nrs, a GCS bucket, or an S3 bucket.
  • get_tensorstore_context() creates a context dict that limits how many threads TensorStore uses for data copying. Without this, TensorStore spawns unlimited threads and can cause out-of-memory errors on the cluster. The limit is read from LSB_DJOB_NUMPROC — the number of LSF-allocated cores (defaults to 1 if not on the cluster).
  • detect_source_order() figures out whether the input data is C-order (row-major) or F-order (column-major). For TensorStore sources, it reads chunk_layout.inner_order directly. For Dask or NumPy arrays, it computes the first chunk and checks the contiguity flags. This determines whether we need a transpose codec in the output spec.

Q: What exactly happens if transpose is placed outside sharding instead of inside?
TensorStore throws "incompatible chunk size constraints" at store creation time. When transpose is an array-level codec, it changes the logical layout of the entire array, but sharding_indexed expects a specific chunk layout for its internal index. Placing transpose inside sharding's inner codec list means it only applies within each inner chunk — the shard index stays in C-order while the data bytes within each chunk are F-order. The error message does not suggest the correct placement — you have to know the Zarr v3 codec pipeline semantics.

Q: Why hardcode 64-cubed inner chunks and 1024-cubed shards?
64-cubed (~512 KB for uint16) is the random-access granularity — a viewer loads one inner chunk at a time, and 512 KB gives sub-second load times while still compressing efficiently with zstd. 1024-cubed shards pack 16×16×16 = 4,096 inner chunks per shard file, reducing file count by 4,096× compared to unsharded — critical on object stores where listing and metadata operations are expensive. The ~2 gigabyte (GB) shard size is within TensorStore's comfortable memory budget for write buffering. These values come from empirical testing with Neuroglancer, WebKnossos, and napari on Janelia datasets.

Q: Why does get_kvstore_spec() support GCS/S3/HTTP if all Janelia data is on local /nrs?
Forward compatibility. The same codebase can write to cloud object stores without code changes — just pass gs://bucket/path instead of /nrs/path. This also enables reading from public HTTP-hosted datasets (like the Allen Institute or OpenOrganelle) without downloading them first.

Q: Why set concurrency to LSF-allocated cores instead of a fixed number?
LSF allocates memory proportional to cores (15 GB/core at Janelia). If you request 4 cores and 60 GB, but TensorStore uses 32 threads, it can exceed both CPU time limits and memory. Matching TensorStore's thread count to the LSF allocation ensures the job stays within its resource envelope.

Q: Why zstd level 5 for sharded and level 1 for non-sharded?
Sharded data is written once and read many times by viewers, so higher compression (level 5) pays off over time. Non-sharded data is typically intermediate (e.g., s0 before pyramid generation) and may be re-read immediately for downsampling, so lower compression (level 1) prioritizes write speed.

metadata_utils.py: OME-NGFF Metadata Management

  • metadata_utils is the largest utility module. It handles all OME-NGFF metadata — creating it, reading it, updating it after downsampling, and pre-creating it before multi-job runs.
  • The top-left code shows what create_zarr3_ome_metadata() actually builds. The structure is three layers deep: zarr_format and node_type at the top, then attributes.ome.multiscales which is an array of datasets. Each dataset has a path (s0, s1, etc.) and coordinateTransformations with scale factors — those are the voxel sizes for that pyramid level. The axes array defines each dimension's name, type (space/time/channel), and unit.
  • The bottom-left code shows how voxel sizes get updated after downsampling. There are two methods, and the choice matters for accuracy in viewers like WebKnossos and Neuroglancer.
  • Cumulative factor (preferred method): during downsampling, the downsampler stores the exact cumulative factor in each level's zarr.json at attributes.custom.cumulative_factor — for example [1, 2, 2] for s1 meaning Z was not halved but Y and X were. To get the voxel size at any level, you just multiply s0's voxel size by the cumulative factor. This gives exact results because the factor is recorded at write time, before any rounding happens.
  • Shape-ratio fallback: when cumulative_factor is missing — for example on older datasets or datasets converted by other tools — the code infers voxel sizes by dividing s0's shape by the level's shape (s0_shape[i] / lvl_shape[i]). This is less accurate because downsampling uses ceiling division (e.g. 1025 / 2 = 513, not 512.5), so the ratio doesn't perfectly match the true downsampling factor. The error is small but compounds over many levels.
  • The right code shows the pre-creation flow. Before submitting 100 LSF jobs, the main process does two things: first it creates all the shard subdirectories, then it writes zarr.json once. After that, the worker jobs only write data chunks — they never touch metadata. Without this, 100 jobs starting at the same time would all try to create zarr.json and corrupt each other's writes.
  • detect_level_format() solves the naming conflict between Janelia tools (s0/s1/s2) and BigStitcher (0/1/2). It checks what already exists on disk and follows that convention.
  • extract_compression_from_zarr3() reads the codec configuration from an existing dataset's zarr.json. When you add s2/s3 levels to a dataset that already has s0/s1, the new levels inherit the same compression settings automatically.
  • translation_transforms() calculates the half-voxel translation offset required by the OME-NGFF spec. After downsampling, the origin shifts by half a voxel at the new resolution, and this function computes that offset for each level's coordinateTransformations.

Q: Why store cumulative_factor in per-level metadata instead of computing it from the pyramid plan?
The pyramid plan is ephemeral — it exists only during the coordinator script's execution and is not persisted. After LSF jobs complete, the only source of truth is what's on disk. The shape-ratio fallback (s0_shape / level_shape) is less accurate because TensorStore uses ceiling division for downsampling (e.g., 1025/2 = 513), so the ratio is 1025/513 = 1.998 instead of the true factor 2.0. Over multiple levels this compounds, producing non-power-of-2 voxel sizes that break WebKnossos. Storing the exact integer factors avoids this entirely.

Q: What is the half-voxel translation offset and why does it matter?
When you downsample 2×, the center of the first output voxel is not at position 0 — it's at 0.5× the new voxel size. The OME-NGFF spec requires coordinateTransformations to include this translation so viewers align all pyramid levels correctly. Without it, zooming in/out in Neuroglancer or napari would show a slight shift between levels.

Q: Why pre-create shard directories instead of letting TensorStore create them on-the-fly?
TensorStore creates directories lazily — the first job to write a chunk in a new shard directory will create it. But if two jobs try to create the same directory simultaneously (race condition), one may fail with a "directory already exists" error depending on the filesystem. Pre-creating all directories from a single process eliminates this entirely.

pyramid_utils.py: Anisotropic Downsampling Algorithm

  • pyramid_utils is pure functions — give it shapes and voxel sizes, it returns a complete plan. No files touched.
  • Start with the credit panel on the top-right. The core anisotropic ratio algorithm comes from Yurii Zubov at Janelia. The link goes directly to his get_downsampling_factors() function. What I built on top was the four stopping rules for knowing when to stop generating levels, a complete pyramid plan pre-calculator with LSF resource estimates, etc.
  • The code on the left shows the core algorithm logic. For each spatial axis, compute its voxel size ratio against all other axes. If all ratios are >= 2.0, that axis is already large relative to the others — don't halve it, it would make the anisotropy worse. Otherwise, halve it. The goal is to make voxel sizes more homogeneous with each downsampling level. This is critical for microscopy data where Z resolution is often 5–10× worse than XY.
  • The example table below shows both array shape and voxel size side by side — shapes shrink while voxel sizes grow. Starting from shape [200, 2048, 2048] with voxels [1000, 160, 160] nanometer (nm), typical confocal with thick Z. The "Input Z/Y" column shows the ratio that drives each decision. At s1 the input Z/Y is 6.3× which is way above the 2× threshold, so Z is skipped — only Y and X are halved. Shapes go to [200, 1024, 1024], voxels to [1000, 320, 320]. At s2 the input Z/Y is still 3.1× above 2×, so Z is skipped again — shapes [200, 512, 512], voxels [1000, 640, 640]. At s3 the input Z/Y drops to 1.6× which is below 2×, so now all three axes are halved — shapes [100, 256, 256], voxels [2000, 1280, 1280]. Z finally gets downsampled because the voxels are approaching isotropic.
  • Key functions. calculate_anisotropic_downsample_factors() is Yurii's core algorithm — it returns per-level factors like [1, 2, 2]. calculate_num_multiscale_levels() wraps it in a loop, calling the algorithm at each level until a stopping rule fires. calculate_pyramid_plan() ties it all together — it predicts shapes, factors, and LSF resources for every level, producing a complete dict that PyramidPlanner uses to submit bsub jobs. resolve_downsample_method() picks mean for images or mode for segmentations — because averaging label 1 and label 2 gives 1.5 which isn't a valid label. It auto-detects by checking the path for keywords like 'label', 'mask', 'seg'.
  • The 4 stopping rules below that. Rule 1: if the output volume would be smaller than one chunk, stop — there's nothing useful to downsample further. Rule 2: if all dimensions are below a threshold like 32, stop. Rule 3 is the most interesting — WebKnossos requires each pyramid level's magnification to be an exact power of 2. Because the anisotropic algorithm skips Z early on, the cumulative magnification can drift. For example, Z's cumulative factor might be 2 while XY's is 8. Rule 3 computes the error: |actual_mag - nearest_power_of_2| / nearest_power_of_2. If any axis drifts more than 4%, the pyramid stops there. Rule 4: safety cap at level 20, a hard limit to prevent infinite loops.
  • At bottom-right — the --per_level_factors flag gives full manual control — semicolon-separated Z,Y,X triples for each level. Useful when auto-anisotropic doesn't produce the exact factors needed. The --single_level_factor flag creates a single pyramid level from s0 with a custom cumulative factor — handy for creating just one specific level without building the full pyramid chain.

Q: How does the 4% magnification drift rule interact with WebKnossos in practice?
WebKnossos requires each pyramid level's cumulative magnification to be an exact power of 2 in all spatial dimensions. The anisotropic algorithm can produce levels where Z has been skipped several times (e.g., Z magnification = 2 while XY = 8). Rule 3 checks |actual_mag - nearest_power_of_2| / nearest_power_of_2 for each axis. The 4% threshold was chosen empirically to allow for ceiling-division rounding (e.g., 1025/2 = 513, not 512) while still catching genuine drift. Without this rule, WebKnossos rejects the dataset with a validation error. The --preset webknossos flag also enforces power-of-2 chunk and shard shapes.

Q: Why use voxel size ratios instead of just always halving all axes?
Always halving all axes (isotropic [2,2,2]) works when voxels are already cubic, but microscopy data is rarely cubic. Confocal Z might be 1000 nm while XY is 160 nm — halving Z when it's already 6× coarser makes the data even more anisotropic. Viewers like Neuroglancer render anisotropic voxels as stretched, so keeping voxel sizes converging toward isotropic gives better visual quality at every zoom level.

Q: What happens if someone passes --per_level_factors that violate WebKnossos power-of-2 requirements?
The manual factors override the anisotropic algorithm entirely — no stopping rules are applied. If the factors produce non-power-of-2 cumulative magnifications, the resulting pyramid will work in Neuroglancer and napari but may be rejected by WebKnossos. Users who need WebKnossos compatibility should use --preset webknossos or let the auto algorithm handle it.

Q: Why does the algorithm use ceiling division (1025/2 = 513) instead of floor division (1025/2 = 512)?
TensorStore's downsample driver uses ceiling division to ensure no source voxels are lost. With floor division, the last row/column of voxels would be silently dropped when dimensions are odd. Ceiling division includes those edge voxels in the last output voxel's average, preserving all data at the cost of slightly uneven block sizes.

Auto-Detection

  • Auto-detection allows the system to figure out everything on its own. Only input + output paths required for a basic conversion.   • Format detection is the entry point — auto_detect() checks the file extension and picks the right reader. N5, Zarr, TIFF, ND2 each get a native reader. Anything else falls through to BioIO as a Tier 3 fallback.
  • Voxel sizes come from each format's own metadata. TIFF reads OME-XML or ImageJ tags, ND2 reads OME metadata, CZI reads Zeiss XML. All get normalized to nanometers.

  • Memory order: Precomputed is always F-order, N5 is always C-order, everything else checks the array flags. Getting this wrong means the transpose codec ends up in the wrong place in the sharding stack.
  • This is what makes batch processing practical — point it at a folder of mixed TIFFs and segmentations, and it handles each one correctly without per-file configuration.

Q: What if auto-detection gets it wrong — do you have to disable all auto-detection?
No. Every property is independently overridable. Pass --voxel_size to fix voxel sizes while everything else still auto-detects. CLI always wins — you never choose between "all auto" and "all manual."

Q: Why path-based keyword matching for labels instead of just checking the dtype?
It checks both. Path keywords catch segmentation stored as uint16 (common for small label sets). Dtype check catches uint32/uint64 even when the path has no keywords. Either signal triggers mode downsampling.

Q: How does the fallback handle conflicting metadata — e.g., TIFF with both OME-XML and ImageJ tags?
OME-XML always wins — the code checks it first. ImageJ tags are only consulted when OME-XML is absent. No merging, no conflict — only one source is ever read.

Q: Precomputed is F-order and N5 is C-order, but what about Zarr or TIFF where it depends on the writer?
The code checks array flags at runtime, not format names. For TensorStore it inspects chunk_layout.inner_order; for Dask/NumPy it checks f_contiguous/c_contiguous. Precomputed and N5 are spec-mandated, but the detection mechanism is the same runtime check for every format.

Resource Auto-Calculation — 3 Output Modes

  • For Auto-Calculation: when you submit a job to the LSF cluster with --submit, the script auto-calculates how much memory, how many cores, and how much wall time to request. The left side shows how it calculates each resource for the three output formats. The right side shows what actually happened when we ran a real 9 GB TIFF through all three.
  • Start with the green table — "How It Calculates." There are three columns because the three output formats have completely different bottlenecks. Zarr3 sharded writes a small number of large shard files — each one is about 2 GB. The bottleneck there is memory, because you need to hold shard buffers. Non-sharded formats — both Zarr2 and Zarr3 with --no_sharding — write thousands of tiny chunk files. The bottleneck is I/O throughput. Different bottleneck, different formula.
  • Cores: For sharded, cores come from memory — ceil(mem/15) × 2, capped at 8. More memory means bigger shards, which benefit from more compression threads. For non-sharded, cores come from dataset size — ceil(size_gb/25) × 2, minimum 4, capped at 8. The 25 means "one extra core pair for every 25 GB of data." A 9 GB dataset gets the minimum 4 cores. A 50 GB dataset gets ceil(50/25) × 2 = 4. A 108 GB dataset gets ceil(108/25) × 2 = 10, but the cap brings it down to 8. We doubled the old value because writing many small files benefits from I/O parallelism.
  • Memory: Sharded memory is driven by shard buffers. The "× 2" is because each shard needs two copies in memory — one for reading the source data and one for writing the compressed output. The "× 3" is because TensorStore can have up to 3 shards in flight at once — one being read, one being compressed, one being flushed to disk. The "base" is a small amount that scales with dataset size — 2 GB for small datasets up to 20 GB for very large ones. The "4 GB" is fixed overhead for Python, Dask, and TensorStore themselves. The whole sum then gets a 1.5× safety margin before rounding up to the nearest 5 GB. Non-sharded memory is much simpler — dataset_size × 0.1, meaning we budget about 10 percent of the dataset size for Dask working memory and intermediate buffers, plus a 2 GB reader buffer for source chunk caching, plus 4 GB task overhead. But then the cluster policy kicks in: minimum 15 GB per core. So if you request 4 cores, the cluster forces at least 60 GB regardless of what the formula says.
  • Wall time: This is where the calculation order matters. We calculate cores first, then wall time, because wall time divides by core count. For sharded: per-shard time (0.5 to 3 minutes depending on shard size) × total shards, divided by cores × 0.85 efficiency, × 2 safety margin. The 0.85 is the parallel efficiency factor — we expect 85 percent of ideal speedup from adding cores. Why not 1.0? Because shards are written sequentially to disk, and there is always some overhead from coordinating between cores, compression threads, and disk I/O. But large shard writes are mostly CPU-bound (compression), so 85 percent is realistic. For non-sharded: the larger of throughput-based and per-chunk estimates, divided by cores × 0.7 efficiency, × 3 safety margin. The 0.7 is lower — only 70 percent efficiency — because writing thousands of tiny files hits filesystem contention on a network drive. Each file needs a metadata operation (create, write, close), and the network filesystem serializes those. More cores help with compression but the disk becomes the bottleneck sooner.
  • Amber table — Adaptive Chunk Sizing. Non-sharded formats need to pick a chunk size. Small datasets under 20 GB get 64³ chunks — fine-grained access for viewers. 20 to 100 GB gets 128³ — 8× fewer files. Over 100 GB gets 256³ — 64× fewer files. Without this, a 108 GB dataset would create 401,000 tiny chunk files and the wall time estimate hits the 96-hour cap. Zarr3 sharded doesn't need this — it always uses shard=1024, inner chunk=64 on spatial axes, because that combination is what Neuroglancer, WebKnossos, and napari all work best with.
  • Why hardcoded 1024 and 64? The 1024 shard gives 16 × 16 × 16 = 4,096 inner chunks per shard — large enough for efficient disk writes, small enough to fit in memory. The 64 inner chunk is the random-access granularity — a viewer loads one 64³ block, not the whole 2 GB shard. Non-spatial axes like channel and time always get 1, so viewers can load a single channel without pulling unrelated data. Making these configurable would add complexity without real benefit.
  • Blue table — Real Benchmark. We ran the same 9 GB TIFF through all three formats on the Janelia cluster. Look at the arrows — requested on the left, actual on the right. Zarr3 sharded finished in 2 minutes. Non-sharded took about 1 hour 50 minutes. That is 55× faster for the same data.
  • Why the gaps between requested and actual? Three reasons.
    — Memory: we requested 60 GB but only used 8.2 GB (sharded) or 400 MB (non-sharded). That is the 15 GB/core cluster policy — 4 cores × 15 = 60, even though the real need is much lower.
    — Cores: we reserved 4 but peak CPU was 2.4 (sharded) or 1.0 (non-sharded). The converter writes chunks one at a time in its main loop. The 2.4 for sharded comes from TensorStore's internal compression threads. The 1.0 for non-sharded means the job is I/O-bound — writing thousands of small files to network filesystem is the bottleneck, not CPU. So why request 4? Two reasons: first, the 15 GB/core memory policy — 4 cores is how we get 60 GB memory allocated. Second, the extra cores are available for burst usage by TensorStore compression threads and Dask source reads, even if sustained CPU stays low.
    — Wall time: we requested 30 minutes but finished in 2 (sharded), requested 3 hours but finished in under 2 (non-sharded). The safety margins — 2× for sharded, 3× for non-sharded — are intentionally generous. A killed job wastes all the work done so far, and the cluster does not charge for unused time.
  • Purple table — Cluster Policy. Three hard rules from Janelia LSF: minimum 15 GB per core, 96-hour wall time cap with a warning, and 8-core cap because these jobs are I/O-bound not CPU-bound. BioIO and BioFormats readers get multipliers — 3× memory and 10× wall time — because Dask-based processing is much slower.
  • CLI override. If any auto-calculated value is wrong for a specific dataset, the user can override with --memory, --wall_time, or --cores. The auto-calculation is a good default, not a hard lock.

Q: Why 0.85 efficiency for sharded but only 0.7 for non-sharded?
Sharded writes are mostly CPU-bound (compressing large 2 GB shards), so adding cores gives close to linear speedup. Non-sharded writes are I/O-bound — thousands of small files each need create/write/close on the network filesystem, which serializes regardless of core count.

Q: Why not just request the minimum resources and let the job run longer?
A killed job loses all progress — there's no checkpointing mid-conversion. Over-requesting is cheap (the cluster doesn't charge for unused time), but under-requesting means starting over from scratch.

Q: Why 15 GB per core minimum — isn't that wasteful for I/O-bound jobs?
That's Janelia's cluster policy, not ours. Each node has a fixed memory-to-core ratio. Requesting 4 cores automatically reserves 60 GB whether you use it or not. The formula accounts for this so the requested values match what the cluster actually allocates.

Q: Could you tune the safety margins and efficiency factors per dataset instead of fixed constants?
Each dataset runs once — there's no history to learn from. The fixed values come from benchmarking across dozens of real datasets. Getting them wrong by 2× is fine (job finishes early); getting them wrong by 0.5× kills the job.

OME-NGFF Compliance & Compatibility

  • This slide shows how to make the output work correctly with viewers and tools.
  • Top left — OME-NGFF Versions table. TensorSwitch writes two flavors. Zarr3 uses the latest v0.5 spec — metadata lives inside zarr.json under the attributes.ome key. Zarr2 uses the older v0.4 spec — metadata lives in a separate .zattrs file with the multiscales key. The "live" links open actual converted datasets on FileGlancer so you can see the real directory structure I recently converted.
  • Translation Transforms — Half-Voxel Offset. In multi-resolution imaging, when you downsample by 2×, each new voxel covers two old voxels. The center of the new voxel is not at position zero — it is shifted by half the difference between the old and new voxel size. The diagram shows it: s0 is 160 nanometers (nm), s1 is 320 nm, so the offset is 0.5 × (320 − 160) = 80 nm. Without this translation, Neuroglancer's pyramid levels are misaligned — zooming in and out causes the image to visibly jump because each level thinks its origin is at position zero. The code is in metadata_utils.py.
  • Unit Handling — Two Rules. This is an intentional design choice, not just following a standard. Rule one: new conversions always output in nanometers. Why? Because nanometers give integer voxel sizes for typical microscopy data — 160 nm, 320 nm, 640 nm. Integers are clean and every tool handles them correctly. If we used micrometers instead, you get 0.16 µm, and some tools round that to 0.2 µm, which silently corrupts your coordinate system. Rule two: when downsampling an existing Zarr, we preserve whatever unit s0 already has. If someone already stored data in micrometers, we do not re-convert — that would make voxel sizes 1000× wrong. The bottom note shows the failure mode: µm gets converted to nm on the first run, then converted again on re-downsample, giving nanometers-of-nanometers.
  • Top right — OMERO Channel Metadata. The --omero flag writes channel names, colors, and display windows into the metadata. OMERO-compatible viewers like napari and OMERO.figure use this to render each channel with the correct look-up table automatically. Without it, viewers show all channels in grayscale or pick random colors.
  • Bottom right — WebKnossos Preset. WebKnossos has strict requirements — inner chunks must be 32³ (that is its block size), shards must be 1024³, output must be Zarr3, and cumulative downsample factors must be exact powers of 2 at every level. The anisotropic algorithm can produce non-power-of-2 factors — for example, Z stays at 1× while XY is at 8×, and WebKnossos rejects that. The --preset webknossos flag sets all four constraints with one argument. The code is in __main__.py.
  • Why presets? Labs that always convert for the same viewer should not have to remember chunk sizes, shard sizes, and factor constraints every time. One flag and the output passes validation. Currently only WebKnossos is implemented, but the pattern supports adding more — a Neuroglancer preset or a BigDataViewer preset could be added the same way.

Q: The slide says OME-NGFF "v0.5" for Zarr3 — is v0.5 actually finalized?
As of early 2026, v0.5 is still technically a draft. The metadata structure TensorSwitch produces (attributes.ome.multiscales in zarr.json) follows the current working draft and is already compatible with Neuroglancer, WebKnossos, and napari. The v0.4 spec for Zarr2 is finalized. If v0.5 changes before finalization, create_zarr3_ome_metadata() in metadata_utils.py would need updating — but the risk is low since the core multiscales structure has been stable across drafts.

Q: Does the half-voxel translation apply to non-spatial axes like channel or time?
No. Translation is only computed for spatial axes (x, y, z). Channel and time have no physical spacing that shifts with downsampling — their scale factors stay 1.

Q: What if a lab needs both WebKnossos and Neuroglancer — do they need two copies?
No. Neuroglancer reads the same Zarr3 sharded format with the same chunk/shard sizes. The main difference is WebKnossos requires exact power-of-2 factors, which Neuroglancer also handles fine. One conversion with --preset webknossos works for both.

Q: Why not just always use nanometers and skip the "preserve s0 unit" rule?
If someone stored data in micrometers and you re-convert to nanometers during downsampling, the voxel sizes become 1000× too large. The preserve rule means downsampling never changes the coordinate system — it only adds new levels.
