Reader Tiers Overview

  • The idea is simple: always use the fastest reader that can handle the format. If TensorStore knows the format natively, use Tier 1 — zero overhead, pure spec dict. If not, fall back down the tiers until something works.
  • Tier 1 is N5, Zarr2, Zarr3, and Neuroglancer Precomputed. These are TensorStore-native — the reader just returns a spec dict and TensorStore opens the file directly. No data conversion, no Dask, just a dict.
  • Tier 2 is the custom readers — TIFF, ND2, IMS, CZI, HDF5. These wrap data as Dask arrays for lazy loading. No data is read until a chunk is actually needed by the converter.
  • Tier 3 is the BIOIO adaptor. Rather than writing a custom reader for every obscure format, TensorSwitch hands off to the bioio ecosystem — install the right bioio plugin and the format just works. LIF, DeltaVision, SLDY, and 20+ more covered this way.
  • Tier 4 is the Java fallback via Bio-Formats 8.5.0. 150+ formats. JVM startup is about 30 seconds, but it's the last resort that almost always works.
  • All readers share the same BaseReader contract. The 3 abstract methods are what every reader must implement. The 3 concrete defaults are provided by BaseReader but any reader can override them — CZIReader overrides get_ome_metadata() to handle the V-to-T axis rename, for example.
  • get_source_info() is worth highlighting — it returns a unified dict covering shape, dtype, dimension names, chunk shape, compression type, and memory order. This is what lets the converter auto-configure output settings without knowing which reader is being used.
  • supports_remote() is False by default because most Tier 2–4 readers work only with local files. Tier 1 readers override this to True because TensorStore natively handles s3://, gs://, and http:// paths.
  • For a detailed look at how tiers differ on real multi-view data, see the CZI case study slide (↓ through Tier 1–4 then one more).

Tier 1: Native TensorStore Readers

  • Tier 1 is the only tier where TensorStore does all the work. The reader just builds a spec dict and hands it over — no Python reads, no numpy, no Dask. TensorStore's C++ engine opens the file and handles all I/O.
  • The driver names in the spec — n5, zarr3, zarr2, neuroglancer_precomputed — are TensorStore's own built-in format backends. We didn't write N5 or Zarr parsers. We just tell TensorStore which driver to use and where the data is, and it handles the rest. This is the whole reason Tier 1 is zero overhead.
  • TensorStore has four core concepts. Driver is the format backend. kvstore is the storage backend — file, GCS, S3, or HTTP. context controls shared resources like thread pools and concurrency limits. Transaction is the atomic write group for concurrent-safe chunk writes.
  • Each Tier 1 reader has its own path-to-kvstore helper. N5 uses _parse_remote_url() defined in n5.py itself — it inspects the path scheme and returns the right kvstore dict: file for local paths, gcs for gs://, s3 for s3://, http for HTTP. Zarr and Precomputed use build_kvstore() from base.py, which does the same thing.
  • The kvstore is the only thing that changes between local and remote — the driver stays 'n5', 'zarr3', etc. That's why Tier 1 supports remote data with no extra code: TensorStore handles the actual I/O once it sees the kvstore spec.
  • Tier 1 is the only tier where supports_remote() returns True. Tier 2, 3, and 4 work with local files only — they wrap data as Dask or Java arrays that don't support remote access natively.
  • ZarrReader detects version by checking which metadata file exists — zarr.json means v3, .zarray means v2. That detection decides which spec to return. Zarr3 gets driver: 'zarr3', Zarr2 gets driver: 'zarr'. One character difference in the spec, but a completely different TensorStore backend.
  • The sharding difference is the big one. Zarr3's zarr3 driver reads the sharding_indexed codec straight from zarr.json and handles it transparently — outer chunks contain multiple inner chunks, each blosc-compressed. Zarr2's zarr driver has no sharding concept at all. You don't write any sharding logic in Python — TensorStore handles it once it sees the codec in the metadata.
  • Multi-scale is handled differently by each format. Zarr3 and Zarr2 use a dataset_path joined into the kvstore path — so s0 becomes part of the file path. N5 puts the dataset path into spec['path'] separately. Precomputed uses scale_index as a number directly in the spec — no path joining at all.
  • PrecomputedReader is for Neuroglancer's precomputed format, common at Janelia for large segmentation volumes. scale_index=0 is the highest resolution, scale_index=1 is 2x downsampled, and so on. TensorStore picks the right scale from the info file automatically.

Tier 2/3 Lazy Loading Pipeline

This slide explains how TensorSwitch reads non-native formats — TIFF, CZI, ND2, and others — without loading the whole file into memory. Dask is the key adapter that decouples input chunk boundaries from output chunk boundaries.

  • Start with the pipeline diagram on the top-left. For TIFF: tifffile.imread with aszarr=True creates a virtual Zarr store — it maps TIFF pages to Zarr chunks without reading any pixel data. da.from_zarr wraps that in a Dask task graph holding only metadata — about 1 MB for a 200 GB file. Nothing has been loaded yet. The reader then wraps the Dask array in a dict with driver='array'. This is NOT a real TensorStore spec — it's just a marker dict so it can go through the same get_tensorstore_spec interface as Tier 1 readers. TensorStore never opens it. The converter detects driver=='array', extracts the Dask array directly, and sets input_store to None. TensorStore is completely bypassed on the read side.
  • Now the code block below the diagram. This shows the 3-step flow of how chunk boundary mismatch is handled. The converter iterates over the OUTPUT chunk grid, not the Dask chunk grid. Dask's .compute() acts as the transparent adapter between mismatched input and output boundaries.
  • Step 1 — Output chunks are the master. get_chunk_domains() in chunk_utils.py (lines 67–94) tiles the entire output space into output-sized chunks — for example, 128 cubed. It is completely independent of the input Dask array's chunk boundaries. The output writer decides the iteration order and chunk size. This completely decouples input format from output format.
  • Step 2 — Convert each output chunk domain to Python slices. _domain_to_slices() in converter.py (lines 522–537) turns a TensorStore IndexDomain into standard Python slices — like (slice(0,128), slice(0,128), slice(0,128)). These are regular Python slices that work on any array, including Dask arrays.
  • Step 3 — Dask handles the stitching. self._dask_array[slices].compute() at converter.py (lines 343–350) is where the magic happens. Dask internally figures out which of its own chunks overlap the requested slice. It loads only the overlapping portions from disk. It automatically stitches them into a single contiguous numpy array. And it returns it — the converter never knows or cares about Dask's internal chunk boundaries. There is no explicit chunk mapping anywhere in the code.
  • Concrete example — the bottom-right amber panel. Say input is a TIFF with 512×512 page-based Dask chunks, and output is Zarr3 with 128 cubed chunks. Request [0:128, 0:128, 0:128] — Dask recognizes one internal chunk covers this, loads only that portion of the TIFF page, extracts the region, done. Request [384:512, 384:512, 384:512] — if it crosses a page boundary, Dask loads 2 or more pages, stitches the overlapping region, returns a clean 128 cubed numpy array. The converter code is identical in both cases — it just calls .compute() and gets back a numpy array. Dask chunks carry their chunk information from tifffile pages, CZI planes, and so on, and when you slice across boundaries, .compute() transparently stitches them. The output chunks don't need to know about input chunks at all — Dask is the adapter layer.
  • Memory efficiency. Each .compute() call loads only the requested slice. Peak memory is one output chunk — about 32 MB — not the whole file. This is why TensorSwitch can convert 200 GB TIFFs on cluster nodes with only 15 GB of RAM. Per-chunk TensorStore transactions on the write side ensure memory doesn't build up during writes either.
  • Future consideration: ts.virtual_chunked? We currently don't use TensorStore's virtual_chunked anywhere — Dask handles all the lazy chunk stitching. But virtual_chunked could be a future enhancement. It lets you create a TensorStore array backed by a Python callback function — so instead of Dask managing input chunks, TensorStore itself would call your function to fill each chunk on demand. This could unify the Tier 1 and Tier 2/3 read paths into a single TensorStore-native pipeline, letting the converter use TensorStore IndexDomain for both native and non-native formats. Right now the Dask approach works well and is simpler, but virtual_chunked is worth exploring if we want tighter TensorStore integration on the read side.
  • The middle-right table shows all the lazy mechanisms per format. CZI uses dask.delayed per plane — the file isn't even opened until .compute(). BioIO's dask_data returns the plugin's lazy array. ND2 uses nd2.to_dask(). IMS and HDF5 wrap h5py datasets with da.from_array(). Every format produces a Dask task graph, not actual data. The converter treats them all identically — iterate output chunks, slice the Dask array, .compute(), write.
  • The top-right comparison table shows the key difference between Tier 1 and Tier 2/3. Tier 1 readers like N5 and Zarr return real TensorStore specs — the converter calls ts.open() and uses TensorStore's native IndexDomain for chunk-aware reading. TensorStore manages both input and output chunks. Tier 2/3 readers bypass TensorStore for reading entirely — Dask manages input chunks, TensorStore manages output chunks, and .compute() is the bridge between them.
  • Q: Why Dask instead of ts.virtual_chunked for Tier 2/3 readers? Every microscopy library (tifffile, nd2, bioio) already produces Dask arrays natively. Using virtual_chunked would require writing custom per-format callbacks that replicate what Dask already does. The tradeoff is TensorStore can't manage input chunks for non-native formats, but Dask's .compute() handles chunk boundary stitching transparently. Migrating to virtual_chunked would unify Tier 1 and Tier 2/3 read paths but adds complexity with no clear performance win today.
  • Q: You said peak memory is "one output chunk ~32 MB" — what's the actual process memory? The 32 MB refers to data only. Actual process memory is: tifffile index (~1 MB for a 200 GB file) + one Dask computation result (~32 MB) + one TensorStore write buffer (~32 MB) + Python/TensorStore overhead (~2–4 GB) = roughly 4–5 GB peak. The 14 GB seen in the CZI benchmark is higher because Dask's task graph metadata for 155,000 delayed tasks takes ~2 GB on its own.

Tier 2: Custom Optimized Readers

These five readers cover the main microscopy formats we see in the lab. They all work the same way — each one wraps the data as a Dask lazy array, hands it to TensorStore as driver: 'array', and nothing actually loads until a chunk is requested.

  • TIFF: tifffile has a trick where you can open a TIFF as a zarr store with aszarr=True. That gives a fully lazy, zero-copy view. For a folder of TIFFs, dask-image globs them and stacks them into one array.
  • ND2 is the simplest — the nd2 library has a built-in to_dask() call. One line. The file handle stays open in the background because Dask needs it alive when pulling chunks later.
  • IMS is HDF5 under the hood, so we walk the tree ourselves: DataSet → ResolutionLevel 0 → TimePoint → Channel → Data. Each channel becomes its own Dask array, then we stack them into CZYX.
  • CZI is the most complex. We build a V × C × Z grid of per-plane dask.delayed reads — each plane opens and closes the file on its own, so it's safe to run in parallel across LSF cores. The [:, :, 0] at the end is because pylibCZIrw returns each plane as a 3D array with shape (Y, X, 1) — there's an extra singleton trailing dimension. We index with 0 to squeeze it down to (Y, X) so Dask can stack the planes cleanly. The actual code has a guard: it only does this when ndim == 3 and shape[2] == 1.
  • HDF5 is the most generic. We try the user-specified path first. If nothing's given, we look for common dataset names, and if that fails too, we just take the largest dataset in the file.
  • Q: CZI opens and closes the file for every single plane — isn't 155,000 open/close cycles expensive? It's necessary because pylibCZIrw file handles are not thread-safe and not picklable, so they can't be shared across Dask workers. The open/close overhead is sub-millisecond — CZI's internal block index makes seeks O(1), not a full file scan. The real I/O cost is ~7 MB per 1920×1920 uint16 plane. The 446 GB CZI conversion completed in 23.2 hours with 14 GB peak memory. The alternative — keeping one handle open — would serialize all reads through one thread, losing all parallelism.

  • Two writers for two ecosystems — Zarr2 for Python tools that haven't moved to Zarr3 yet, N5 for the Java tools at Janelia.

DistributedConverter

  • This is the conversion engine. It's format-agnostic — it only works with TensorStore specs. The reader gives it an input, the writer gives it an output, and it copies chunk by chunk.
  • The code on the left walks through the four steps. First, open the input — Tier 1 readers like N5 and Zarr give back native TensorStore specs. Tier 2 readers like TIFF and CZI give back a Dask array instead. The converter handles both.
  • Second, create the output by calling the writer's create_output_spec(). The writer handles everything format-specific — sharding, codecs, chunk layout.
  • Third and fourth, iterate over chunks and write each one inside a TensorStore transaction. That means if a job crashes halfway, you never get a corrupted partial chunk in the output. Other jobs writing to different chunks keep going independently.
  • The CLI examples below show the two modes. Single-job runs locally or on the cluster with --submit — same code, just wrapped in bsub. Multi-job LSF splits the dataset into chunk ranges with --start_idx and --stop_idx, so you can run 10 or 100 jobs in parallel on different parts of the data.
  • On the right — key features. Transaction-per-chunk is what makes multi-job safe: each chunk write either fully completes or rolls back, so 100+ parallel jobs can write to the same Zarr without corruption.
  • By default the converter preserves source dimensions — 3D in, 3D out. No unwanted singleton axes added. If you need 5D TCZYX output for legacy tools, there's an --expand-to-5d flag, but it's off by default per OME-NGFF RFC-3.
  • It also detects and preserves memory order — if the source is F-order (column-major, like Neuroglancer Precomputed), the output stays F-order. You can override with --force_c_order or --force_f_order.
  • Singleton channel squeeze handles a Precomputed quirk: TensorStore's precomputed driver always adds a 'channel' axis even when there's only one channel, turning a 3D volume into [Z, Y, X, Channel=1]. The converter detects that the channel dimension is size 1, squeezes it out, and writes clean 3D output.
  • Q: What happens when multiple LSF jobs write to the same shard file simultaneously? TensorStore's sharding_indexed codec handles this safely — each inner chunk has its own byte-range entry in the shard index, and writes are serialized at the shard level by TensorStore's internal locking and transaction mechanism. The per-chunk ts.Transaction ensures each chunk write is atomic. Two jobs with adjacent chunk ranges may target different inner chunks in the same shard, and TensorStore handles this correctly on POSIX filesystems like /nrs. Could be a problem on object stores without strong consistency.
  • Q: What happens when a chunk write fails mid-conversion? The current strategy is warn-and-skip: the exception is printed as a warning and the loop continues. The return dict includes chunks_processed which will be less than total_chunks if any failed. Transaction-per-chunk means partial writes are impossible — a chunk is either fully written or not at all. The rationale is one corrupt source region should not abort a TB-scale conversion. There is no post-hoc verification step yet — a --verify mode that reads back chunk metadata would strengthen this.

Downsampler: How Downsampling Actually Works

  • The code on the left shows the three steps. The key is step 2 — TensorStore's downsample driver. You don't manually read blocks of voxels and average them. Instead you open a virtual TensorStore that looks like the downsampled data already exists. When you read a chunk from it, TensorStore fetches the corresponding source region and reduces it on the fly.
  • Step 3 just copies chunks from the virtual view to the output store, using the same transaction-per-chunk pattern as DistributedConverter. That's why multi-job LSF works here too — different jobs copy different chunk ranges.
  • Six methods on the right. Mean averages voxel intensities — correct for microscopy images. Mode picks the most common value in each region — correct for segmentation labels, because averaging label 1 and label 2 would give 1.5, which isn't a valid label. Stride just picks every Nth voxel — fastest but introduces aliasing.
  • Auto-detect checks the input path for keywords like 'label', 'mask', 'seg'. If found, it uses mode. Otherwise mean. So running --auto_multiscale on .../labels/segmentation/s0 automatically picks mode without you having to specify it.
  • Cumulative factors on the bottom right. In a pyramid, each level has a per-level factor — how much to reduce from the previous level. For example, s0→s1 is [1,2,2] (skip Z, halve Y and X), and s1→s2 is also [1,2,2]. The cumulative factor multiplies these together: for s2 it's [1, 2×2, 2×2] = [1, 4, 4]. This single cumulative factor tells the downsample driver exactly how much to reduce from s0 in one step. The Downsampler passes this directly to the TensorStore downsample spec so it can read from s0 (or the previous level in chained mode) and produce the correct output in a single pass.

PyramidPlanner: Chained Downsampling — Key Innovation

  • The chain diagram at the top is the key idea — each level reads from the previous level, not from s0. This is the architecture that makes deep pyramids practical on the cluster.
  • The Python code on the left is the core loop inside _generate_chained_coordinator_script(). It iterates over each level in the pyramid plan. The critical line is where source is set to the previous level — level minus one, not s0. Then it auto-calculates memory based on that source shape. Because the source shrinks at every level, later levels need less memory, less wall time, fewer cores.
  • The bash on the right is what that loop generates — a coordinator script. The submit_and_wait function submits a downsample job via bsub, then calls bwait to block until it finishes. Only then does the next level get submitted. Simple sequential script, no complex LSF dependency graphs.
  • Notice the decreasing resources in the bash — 30G for s1, 15G for s2, 8G for s3. And the wall time drops too — 1:00, 0:30, 0:15. That's the direct benefit of chaining — each level's source is smaller, so it needs less of everything.
  • The Auto-Calculated Resources panel shows the actual formulas. Memory is based on shard size times the downsample factor — that's how much source data you need to read per output shard — plus write buffer and overhead, with a 1.5× safety margin, clamped between 8 and 128 GB. Wall time uses empirical shard-size tiers: small shards under 0.1 GB take about half a minute each, medium shards up to 1 GB take about 2 minutes, and large shards over 1 GB take about 4 minutes. Multiply by number of shards, add read overhead based on source size, then apply 2× safety margin. The range is 15 minutes to 12 hours.
  • Cores are derived from memory — divide by 15 and double, capped at 8. We cap at 8 because this is I/O-bound work — more cores don't help. The Janelia cluster enforces 15 GB per core minimum, so if you request 6 cores you get at least 90 GB allocated. The code pre-calculates this so our request matches what the cluster actually gives us.
  • Why chain? Look at the diagram on the bottom right. My first attempt had all levels read directly from s0 — the fan-out pattern. For s4, that meant 2048× read amplification per output voxel. The s4 job ran over 4 hours and never finished. I switched to chained, where each level reads only the previous level. That gives constant 8× amplification per level. The CZI dataset — 446 GB, 6 levels — completed in 73 minutes total.
  • After all levels complete, the coordinator updates the root OME-NGFF metadata with all the pyramid level paths. This runs last to avoid race conditions.
  • Q: Doesn't chaining introduce compounding quality loss from repeated downsampling? In theory, all-from-s0 is more mathematically correct. In practice, the quality difference is negligible for microscopy data — averaging 8 voxels (2×2×2 from previous level) versus averaging 2048 voxels (from s0) produces differences below the noise floor of the microscope. The performance difference is not negligible — the CZI dataset completed in 73 minutes chained versus never finishing from s0 due to 2048× read amplification at s4.
  • Q: What happens if the coordinator script is killed between bwait calls? Any currently-running LSF job continues to completion — it's independent of the coordinator. But subsequent levels are never submitted. The data already written is valid — each level is a complete, self-contained Zarr array. To recover, re-run --auto_multiscale with --skip_existing, which checks for existing zarr.json files and only submits missing levels. The current bwait approach was chosen over LSF job dependencies (bsub -w "done(JOB_ID)") for simplicity and debuggability — you can watch the coordinator's stdout in real time.

tensorstore_utils.py: TensorStore Spec Builder

This module is the single source of truth for building TensorStore specs. Every writer calls into here — Zarr3Writer, Zarr2Writer, N5Writer — none of them construct specs from scratch.

  • The left code block shows the trickiest part. When you use F-order (Fortran/column-major) with sharding, the transpose codec has to go INSIDE the sharding_indexed codec's inner_codecs list — before bytes and compression. If you put it at the array level instead, TensorStore throws an "incompatible chunk size constraints" error. This was a real debugging headache because the error message doesn't tell you what's wrong — you just have to know that transpose belongs inside sharding.
  • The right code block shows NON_SPATIAL_AXES and build_default_shape(). By default, channels, time, and view axes get chunk size 1. The reason is most microscopy workflows view one channel or one time point at a time — if chunk size is 1, loading a single time point reads exactly one chunk. If it were 5, you'd pull in 5 time points just to see one. Spatial axes (z, y, x) get a larger size, typically 64 for inner chunks and 1024 for shards. This is just the default though — users can override it with --chunk_shape on the CLI, for example --chunk_shape 2,2,64,64,64 to group 2 channels per chunk. There's no validation preventing it.
  • zarr3_store_spec() builds the full TensorStore JSON spec for Zarr3 output — kvstore path, data type, chunk grid, sharding_indexed codec with inner codecs, and compression. It's the most complex function because Zarr3 sharding has a nested codec structure.
  • zarr2_store_spec() does the same for Zarr2 but simpler — no sharding, flat codec list. The key detail is fill_value: 0 instead of null, which is required for Neuroglancer compatibility. Neuroglancer can't display arrays with null fill values.
  • get_kvstore_spec() abstracts storage backends. You pass it a path and it figures out the right driver — "file" for local paths, "gcs" for gs:// URLs, "s3" for s3:// URLs, "http" for http/https URLs. This means the same spec-building code works whether you're writing to /nrs, a GCS bucket, or an S3 bucket.
  • get_tensorstore_context() creates a context dict that limits how many threads TensorStore uses for data copying. Without this, TensorStore spawns unlimited threads and can cause out-of-memory errors on the cluster. The limit is set to the number of LSF-allocated cores.
  • detect_source_order() figures out whether the input data is C-order (row-major) or F-order (column-major). For TensorStore sources, it reads chunk_layout.inner_order directly. For Dask or NumPy arrays, it computes the first chunk and checks the contiguity flags. This determines whether we need a transpose codec in the output spec.
  • Q: What exactly happens if transpose is placed outside sharding instead of inside? TensorStore throws "incompatible chunk size constraints" at store creation time. When transpose is an array-level codec, it changes the logical layout of the entire array, but sharding_indexed expects a specific chunk layout for its internal index. Placing transpose inside sharding's inner codec list means it only applies within each inner chunk — the shard index stays in C-order while the data bytes within each chunk are F-order. The error message does not suggest the correct placement — you have to know the Zarr v3 codec pipeline semantics.
  • Q: Why hardcode 64-cubed inner chunks and 1024-cubed shards? 64-cubed (~512 KB for uint16) is the random-access granularity — a viewer loads one inner chunk at a time, and 512 KB gives sub-second load times while still compressing efficiently with zstd. 1024-cubed shards pack 16×16×16 = 4,096 inner chunks per shard file, reducing file count by 4096× compared to unsharded — critical on object stores where listing and metadata operations are expensive. The 2 GB shard size is within TensorStore's comfortable memory budget for write buffering. These values come from empirical testing with Neuroglancer, WebKnossos, and napari on Janelia datasets.

metadata_utils.py: OME-NGFF Metadata Management

This is the largest utility module. It handles all OME-NGFF metadata — creating it, reading it, updating it after downsampling, and pre-creating it before multi-job runs.

  • The top-left code block shows what create_zarr3_ome_metadata() actually builds. The structure is three layers deep: zarr_format and node_type at the top, then attributes.ome.multiscales which is an array of datasets. Each dataset has a path (s0, s1, etc.) and coordinateTransformations with scale factors — those are the voxel sizes for that pyramid level. The axes array defines each dimension's name, type (space/time/channel), and unit.
  • The bottom-left code block shows how voxel sizes get updated after downsampling. There are two methods, and the choice matters for accuracy in viewers like WebKnossos and Neuroglancer.
  • Cumulative factor (preferred method): during downsampling, the downsampler stores the exact cumulative factor in each level's zarr.json at attributes.custom.cumulative_factor — for example [1, 2, 2] for s1 meaning Z was not halved but Y and X were. To get the voxel size at any level, you just multiply s0's voxel size by the cumulative factor. This gives exact results because the factor is recorded at write time, before any rounding happens.
  • Shape-ratio fallback: when cumulative_factor is missing — for example on older datasets or datasets converted by other tools — the code infers voxel sizes by dividing s0's shape by the level's shape (s0_shape[i] / lvl_shape[i]). This is less accurate because downsampling uses ceiling division (e.g. 1025 / 2 = 513, not 512.5), so the ratio doesn't perfectly match the true downsampling factor. The error is small but compounds over many levels.
  • The right code block shows the pre-creation flow. Before submitting 100 LSF jobs, the main process does two things: first it creates all the shard subdirectories, then it writes zarr.json once. After that, the worker jobs only write data chunks — they never touch metadata. Without this, 100 jobs starting at the same time would all try to create zarr.json and corrupt each other's writes.
  • detect_level_format() solves the naming conflict between Janelia tools (s0/s1/s2) and BigStitcher (0/1/2). It checks what already exists on disk and follows that convention.
  • extract_compression_from_zarr3() reads the codec configuration from an existing dataset's zarr.json. When you add s2/s3 levels to a dataset that already has s0/s1, the new levels inherit the same compression settings automatically — no need to specify --compression.
  • translation_transforms() calculates the half-voxel translation offset required by the OME-NGFF spec. After downsampling, the origin shifts by half a voxel at the new resolution, and this function computes that offset for each level's coordinateTransformations.
  • Q: Why store cumulative_factor in per-level metadata instead of computing it from the pyramid plan? The pyramid plan is ephemeral — it exists only during the coordinator script's execution and is not persisted. After LSF jobs complete, the only source of truth is what's on disk. The shape-ratio fallback (s0_shape / level_shape) is less accurate because TensorStore uses ceiling division for downsampling (e.g., 1025/2 = 513), so the ratio is 1025/513 = 1.998 instead of the true factor 2.0. Over multiple levels this compounds, producing non-power-of-2 voxel sizes that break WebKnossos. Storing the exact integer factors avoids this entirely.

pyramid_utils.py: Anisotropic Downsampling Algorithm

This module is the math behind pyramid generation. It's pure functions — give it shapes and voxel sizes, it returns a complete plan. No files touched.

  • Start with the credit panel on the top-right. The core anisotropic ratio algorithm comes from Yurii Zubov at Janelia CellMap, originally in janelia-cellmap/zarrify. The link goes directly to his get_downsampling_factors() function. What I built on top: four stopping rules for knowing when to stop generating levels, a complete pyramid plan pre-calculator with LSF resource estimates, automatic mean vs mode selection for image vs segmentation data, and WebKnossos power-of-2 validation.
  • The code on the left shows the core algorithm logic. For each spatial axis, compute its voxel size ratio against all other axes. If all ratios are >= 2.0, that axis is already large relative to the others — don't halve it, it would make the anisotropy worse. Otherwise, halve it. The goal is to make voxel sizes more homogeneous with each downsampling level. This is critical for microscopy data where Z resolution is often 5-10x worse than XY.
  • The example table below the code shows both array shape and voxel size side by side — shapes shrink while voxel sizes grow. Starting from shape [200, 2048, 2048] with voxels [1000, 160, 160] nm, typical confocal with thick Z. The "Input Z/Y" column shows the ratio that drives each decision. At s1 the input Z/Y is 6.3× which is way above the 2× threshold, so Z is skipped — only Y and X are halved. Shapes go to [200, 1024, 1024], voxels to [1000, 320, 320]. At s2 the input Z/Y is still 3.1× above 2×, so Z is skipped again — shapes [200, 512, 512], voxels [1000, 640, 640]. At s3 the input Z/Y drops to 1.6× which is below 2×, so now all three axes are halved — shapes [100, 256, 256], voxels [2000, 1280, 1280]. Z finally gets downsampled because the voxels are approaching isotropic.
  • Key functions on the right. calculate_anisotropic_downsample_factors() is Yurii's core algorithm — it returns per-level factors like [1, 2, 2]. calculate_num_multiscale_levels() wraps it in a loop, calling the algorithm at each level until a stopping rule fires. calculate_pyramid_plan() ties it all together — it predicts shapes, factors, and LSF resources for every level, producing a complete dict that PyramidPlanner uses to submit bsub jobs. resolve_downsample_method() picks mean for images or mode for segmentations — because averaging label 1 and label 2 gives 1.5 which isn't a valid label. It auto-detects by checking the path for keywords like 'label', 'mask', 'seg'.
  • The 4 stopping rules below that. Rule 1: if the output volume would be smaller than one chunk, stop — there's nothing useful to downsample further. Rule 2: if all dimensions are below a threshold like 32, stop. Rule 3 is the most interesting — WebKnossos requires each pyramid level's magnification to be an exact power of 2. Because the anisotropic algorithm skips Z early on, the cumulative magnification can drift. For example, Z's cumulative factor might be 2 while XY's is 8. Rule 3 computes the error: |actual_mag - nearest_power_of_2| / nearest_power_of_2. If any axis drifts more than 4%, the pyramid stops there. Rule 4: safety cap at level 20, a hard limit to prevent infinite loops.
  • Bottom-right — per-level and single-level factor overrides. The --per_level_factors flag gives full manual control — semicolon-separated Z,Y,X triples for each level. Useful when auto-anisotropic doesn't produce the exact factors needed. The --single_level_factor flag creates a single pyramid level from s0 with a custom cumulative factor — handy for creating just one specific level without building the full pyramid chain.
  • Q: How does the 4% magnification drift rule interact with WebKnossos in practice? WebKnossos requires each pyramid level's cumulative magnification to be an exact power of 2 in all spatial dimensions. The anisotropic algorithm can produce levels where Z has been skipped several times (e.g., Z magnification = 2 while XY = 8). Rule 3 checks |actual_mag - nearest_power_of_2| / nearest_power_of_2 for each axis. The 4% threshold was chosen empirically to allow for ceiling-division rounding (e.g., 1025/2 = 513, not 512) while still catching genuine drift. Without this rule, WebKnossos rejects the dataset with a validation error. The --preset webknossos flag also enforces power-of-2 chunk and shard shapes.

Resource Auto-Calculation — 3 Output Modes

  • What this slide shows. When you submit a job to the LSF cluster with --submit, the script auto-calculates how much memory, how many cores, and how much wall time to request. The left side shows how it calculates each resource for the three output formats. The right side shows what actually happened when we ran a real 9 GB TIFF through all three.
  • Start with the green table — "How It Calculates." There are three columns because the three output formats have completely different bottlenecks. Zarr3 sharded writes a small number of large shard files — each one is about 2 GB. The bottleneck there is memory, because you need to hold shard buffers. Non-sharded formats — both Zarr2 and Zarr3 with --no_sharding — write thousands of tiny chunk files. The bottleneck is I/O throughput. Different bottleneck, different formula.
  • Cores row. For sharded, cores come from memory — ceil(mem/15) times 2, capped at 8. More memory means bigger shards, which benefit from more compression threads. For non-sharded, cores come from dataset size — ceil(size_gb/25) times 2, minimum 4, capped at 8. The 25 means "one extra core pair for every 25 GB of data." A 9 GB dataset gets the minimum 4 cores. A 50 GB dataset gets ceil(50/25) times 2 equals 4. A 108 GB dataset gets ceil(108/25) times 2 equals 10, but the cap brings it down to 8. We doubled the old value because writing many small files benefits from I/O parallelism.
  • Memory row. Sharded memory is driven by shard buffers. The "times 2" is because each shard needs two copies in memory — one for reading the source data and one for writing the compressed output. The "times 3" is because TensorStore can have up to 3 shards in flight at once — one being read, one being compressed, one being flushed to disk. The "base" is a small amount that scales with dataset size — 2 GB for small datasets up to 20 GB for very large ones. The "4 GB" is fixed overhead for Python, Dask, and TensorStore themselves. Non-sharded memory is much simpler — dataset_size times 0.1, meaning we budget about 10 percent of the dataset size for Dask working memory and intermediate buffers, plus a 2 GB reader buffer for source chunk caching, plus 4 GB task overhead. But then the cluster policy kicks in: minimum 15 GB per core. So if you request 4 cores, the cluster forces at least 60 GB regardless of what the formula says.
  • Wall time row. This is where the calculation order matters. We calculate cores first, then wall time, because wall time divides by core count. For sharded: per-shard time (0.5 to 3 minutes depending on shard size) times total shards, divided by cores times 0.85 efficiency, times 2 safety margin. The 0.85 is the parallel efficiency factor — it means we expect 85 percent of ideal speedup from adding cores. Why not 1.0? Because shards are written sequentially to disk, and there is always some overhead from coordinating between cores, compression threads, and disk I/O. But large shard writes are mostly CPU-bound (compression), so 85 percent is realistic. For non-sharded: the larger of throughput-based and per-chunk estimates, divided by cores times 0.7 efficiency, times 3 safety margin. The 0.7 is lower — only 70 percent efficiency — because writing thousands of tiny files hits filesystem contention on a network drive. Each file needs a metadata operation (create, write, close), and the network filesystem serializes those. More cores help with compression but the disk becomes the bottleneck sooner.
  • Amber table — Adaptive Chunk Sizing. Non-sharded formats need to pick a chunk size. Small datasets under 20 GB get 64-cubed chunks — fine-grained access for viewers. 20 to 100 GB gets 128 — 8 times fewer files. Over 100 GB gets 256 — 64 times fewer files. Without this, a 108 GB dataset would create 401,000 tiny chunk files and the wall time estimate hits the 96-hour cap. Zarr3 sharded doesn't need this — it always uses shard=1024 and inner chunk=64 on spatial axes, because that combination is what Neuroglancer, WebKnossos, and napari all work best with.
  • Why hardcoded 1024 and 64? The 1024 shard gives 16 times 16 times 16 equals 4,096 inner chunks per shard — large enough for efficient disk writes, small enough to fit in memory. The 64 inner chunk is the random-access granularity — a viewer loads one 64-cubed block, not the whole 2 GB shard. Non-spatial axes like channel and time always get 1, so viewers can load a single channel without pulling unrelated data. Making these configurable would add complexity without real benefit.

  • Why the gaps between requested and actual? Three reasons.
    — Memory: we requested 60 GB but only used 8.2 GB (sharded) or 400 MB (non-sharded). That is the 15 GB per core cluster policy — 4 cores times 15 equals 60, even though the real need is much lower.
    — Cores: we reserved 4 but peak CPU was 2.4 (sharded) or 1.0 (non-sharded). The converter writes chunks one at a time in its main loop. The 2.4 for sharded comes from TensorStore's internal compression threads. The 1.0 for non-sharded means the job is I/O-bound — writing thousands of small files to network filesystem is the bottleneck, not CPU. So why request 4? Two reasons: first, the 15 GB/core memory policy — 4 cores is how we get 60 GB memory allocated. Second, the extra cores are available for burst usage by TensorStore compression threads and Dask source reads, even if sustained CPU stays low.
    — Wall time: we requested 30 minutes but finished in 2 (sharded), requested 3 hours but finished in under 2 (non-sharded). The safety margins — 2x for sharded, 3x for non-sharded — are intentionally generous. A killed job wastes all the work done so far, and the cluster does not charge for unused time.

BioIO and BioFormats readers get multipliers — 3x memory and 10x wall time — because Dask-based processing is much slower.

  • Translation Transforms — Half-Voxel Offset. This is a common gotcha in multi-resolution imaging. When you downsample by 2x, each new voxel covers two old voxels. The center of the new voxel is not at position zero — it is shifted by half the difference between the old and new voxel size. The diagram shows it: s0 is 160 nm, s1 is 320 nm, so the offset is 0.5 times (320 minus 160) equals 80 nm. Without this translation, Neuroglancer's pyramid levels are misaligned — zooming in and out causes the image to visibly jump because each level thinks its origin is at position zero. The code is in metadata_utils.py at line 458.
  • Unit Handling — Two Rules. This is an intentional design choice, not just following a standard. Rule one: new conversions always output in nanometers. Why? Because nanometers give integer voxel sizes for typical microscopy data — 160 nm, 320 nm, 640 nm. Integers are clean and every tool handles them correctly. If we used micrometers instead, you get 0.16 µm, and some tools round that to 0.2 µm, which silently corrupts your coordinate system. Rule two: when downsampling an existing Zarr, we preserve whatever unit s0 already has. If someone already stored data in micrometers, we do not re-convert — that would make voxel sizes 1000x wrong. The bottom note shows the failure mode: µm gets converted to nm on the first run, then converted again on re-downsample, giving nanometers-of-nanometers.
  • Top right — OMERO Channel Metadata. The --omero flag writes channel names, colors, and display windows into the metadata. OMERO-compatible viewers like napari and OMERO.figure use this to render each channel with the correct look-up table automatically. Without it, viewers show all channels in grayscale or pick random colors.
  • Bottom right — WebKnossos Preset. WebKnossos has strict requirements — inner chunks must be 32 cubed (that is its block size), shards must be 1024 cubed, output must be Zarr3, and cumulative downsample factors must be exact powers of 2 at every level. The anisotropic algorithm can produce non-power-of-2 factors — for example, Z stays at 1x while XY is at 8x, and WebKnossos rejects that. The --preset webknossos flag sets all four constraints with one argument. The code is in __main__.py at line 218.
  • Why presets? Labs that always convert for the same viewer should not have to remember chunk sizes, shard sizes, and factor constraints every time. One flag and the output passes validation. Currently only WebKnossos is implemented, but the pattern supports adding more — a Neuroglancer preset or a BigDataViewer preset could be added the same way.

  • Q: The slide says OME-NGFF "v0.5" for Zarr3 — is v0.5 actually finalized? As of early 2026, OME-NGFF v0.5 for Zarr3 is still technically a draft/pre-release specification. The metadata structure TensorSwitch produces (attributes.ome.multiscales in zarr.json) follows the current working draft and is compatible with Neuroglancer, WebKnossos, and napari. The v0.4 spec for Zarr2 is finalized. If v0.5 changes before finalization, the create_zarr3_ome_metadata() function in metadata_utils.py would need updating. The risk is low since the core multiscales structure has been stable across drafts and major viewers already consume it.

Dependencies — 11 Critical Packages

  • What this slide shows. The full dependency list split into two groups. Left side is what every install needs. Right side is optional — you only pull these in if you use the corresponding reader tier.
  • Core Dependencies — left table. TensorStore is the foundation — all array reads and writes go through it. Version 0.1.79 or higher is required for the sharding_indexed codec and Zarr v3 support. Dask handles lazy loading for Tier 2 readers like TIFF and ND2 — the array is a task graph that only gets materialized chunk by chunk. Zarr 3.1.1 is the Python Zarr v3 library. NumPy, tifffile, and nd2 are the format-specific readers.
  • Optional / Tier-Specific — right table. h5py is for IMS (Imaris) and HDF5 files. pylibCZIrw is Zeiss's official CZI library — version 5.1.1 has the multi-view support needed for large CZI datasets like the Keller lab's 36-view data. bioio is the Tier 3 plugin framework — it auto-detects which bioio plugin to use for a given file. bioio-bioformats wraps Bio-Formats via a Java bridge for the 150 plus formats in Tier 4. scyjava manages the JVM lifecycle — starting Java, classpath, and cleanup.
  • Standard Library — bottom right. These are all from Python's standard library, no install needed. os, sys, json, glob for file operations. shlex and subprocess for LSF job submission. argparse for CLI parsing. pathlib, urllib, abc, and dataclasses for internal structure.
  • Version pinning. All versions in pyproject.toml use lower bounds with upper caps — for example tensorstore is pinned to >=0.1.79 but less than 0.2. This prevents breaking changes from major version bumps while still allowing patch updates. Pixi locks the exact versions in pixi.lock for reproducibility on the cluster.
